# **Android 与 Java 核心知识点 100 题精选（含答案解析）**

## **第一部分：Java 基础与 JVM（第 1-25 题）**

#### **1\. JVM 的运行时数据区包含哪些部分？**

* **答案：** 包含程序计数器（Program Counter Register）、Java 虚拟机栈（JVM Stack）、本地方法栈（Native Method Stack）、Java 堆（Heap）、方法区（Method Area）以及运行时常量池。

#### **2\. JVM 中哪些内存区域是线程私有的，哪些是线程共享的？**

* **答案：** 线程私有：程序计数器、虚拟机栈、本地方法栈。线程共享：Java 堆、方法区。

#### **3\. 什么是 JVM 的类加载机制？**

* **答案：** 类加载机制是虚拟机把描述类的数据从 Class 文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成可以被虚拟机直接使用的 Java 类型的过程。

#### **4\. 请简述双亲委派模型的工作原理。**

* **答案：** 当一个类加载器收到类加载请求时，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成，每一个层次的类加载器都是如此。因此，所有的加载请求最终都应该传送到顶层的启动类加载器中，只有当父加载器反馈自己无法完成这个加载请求时，子加载器才会尝试自己去加载。

#### **5\. 双亲委派模型有什么作用？**

* **答案：** 保证 Java 核心库的类型安全（例如防止用户自定义一个 java.lang.Object 类并覆盖核心库）；避免类的重复加载。

#### **6\. 强引用、软引用、弱引用、虚引用的区别是什么？**

* **答案：** \* 强引用（StrongReference）：垃圾回收器绝不回收，内存不足时抛出 OOM。  
  * 软引用（SoftReference）：内存充足时不回收，内存不足时在 OOM 前回收。  
  * 弱引用（WeakReference）：只要垃圾回收器运行，不论内存是否充足都会回收。  
  * 虚引用（PhantomReference）：形同虚设，主要用于在对象被回收时收到一个系统通知。

#### **7\. 垃圾收集算法中，“标记-清除”算法有什么缺点？**

* **答案：** 效率问题（标记和清除两个过程的效率都不高）；空间问题（清除之后会产生大量不连续的内存碎片）。

#### **8\. “标记-复制”算法和“标记-整理”算法有什么区别？**

* **答案：** 复制算法将内存分为两块，每次只使用一块，回收时将存活对象复制到另一块，解决了碎片问题但浪费了一半空间，适合新生代；整理算法让所有存活对象都向一端移动，然后直接清理掉端边界以外的内存，适合老年代。

#### **9\. 什么是 Java 的 G1 垃圾收集器？**

* **答案：** G1（Garbage-First）是一款面向服务端应用的垃圾收集器。它将堆内存划分成多个大小相等的独立区域（Region），全面采用分代收集，并能建立可预测的停顿时间模型。

#### **10\. static 关键字的作用是什么？**

* **答案：** 用于修饰成员变量和成员方法，使其属于类而不是属于某个具体的对象；还可用于静态代码块（在类加载时执行且仅执行一次）和静态内部类。

#### **11\. final 关键字修饰类、方法、变量时分别有什么效果？**

* **答案：** 修饰类时，该类不能被继承；修饰方法时，该方法不能被子类重写（Override）；修饰变量时，如果是基本类型则数值不能改变，如果是引用类型则引用地址不能改变（但对象内部的属性可以改变）。

#### **12\. String、StringBuffer 和 StringBuilder 的区别是什么？**

* **答案：** String 是不可变的字符序列；StringBuffer 和 StringBuilder 是可变的。其中 StringBuffer 是线程安全的（方法有 synchronized 修饰），而 StringBuilder 是非线程安全的，但性能更高。

#### **13\. Java 中异常处理的 try-catch-finally 中，如果 try 里有 return，finally 还会执行吗？**

* **答案：** 会执行。finally 块中的代码会在 try 中的 return 语句执行完、但返回值还没有传递给调用者之前执行。

#### **14\. 内部类为什么能访问外部类的私有属性？**

* **答案：** 编译器在编译内部类时，会自动为内部类生成一个指向外部类对象的引用（通常命名为 this$0），并通过外部类自动生成的静态辅助方法来访问私有属性。

#### **15\. 什么是匿名内部类？使用它有什么限制？**

* **答案：** 匿名内部类是没有名字的内部类，通常用于简化代码（如实现事件监听器）。限制包括：必须继承一个父类或实现一个接口；不能有构造方法；只能访问外部的 final 局部变量（Java 8 之后要求隐式 final）。

#### **16\. Java 语言的三大特性是什么？**

* **答案：** 封装（Hide implementation details）、继承（Reuse and extend code）、多态（Dynamic binding / Run-time polymorphism）。

#### **17\. 什么是重写（Override）和重载（Overload）？**

* **答案：** 重写发生在父子类之间，方法名、参数列表和返回值必须相同，是运行时多态；重载发生在同一个类中，方法名相同但参数列表不同（类型、个数、顺序），是编译时多态。

#### **18\. 抽象类（Abstract Class）和接口（Interface）的核心区别是什么？**

* **答案：** 抽象类是对事物的抽象（Is-A 关系），可以包含成员变量和具体实现；接口是对行为的抽象（Like-A 关系/能力契约），在 Java 8 之前只能有抽象方法和常量。一个类只能继承一个抽象类，但可以实现多个接口。

#### **19\. HashMap 的底层数据结构在 JDK 1.8 中发生了什么变化？**

* **答案：** JDK 1.7 采用“数组+链表”；JDK 1.8 引入了红黑树，演变为“数组+链表+红黑树”。当链表长度大于 8 且数组容量大于等于 64 时，链表会转换为红黑树，以提高查询效率（从 $O(n)$ 降到 $O(\\log n)$）。

#### **20\. HashMap 为什么不是线程安全的？**

* **答案：** 在多线程并发环境下，进行 put 操作可能会导致数据覆盖；在 JDK 1.7 中并发扩容还可能会引起链表死循环（死链问题）。

#### **21\. 如何在多线程环境中使用安全的 Map？**

* **答案：** 可以使用 Hashtable（效率低，全表锁），或者使用 Collections.synchronizedMap()，但推荐使用 ConcurrentHashMap（分段锁/CAS+synchronized，高并发下性能好）。

#### **22\. ArrayList 和 LinkedList 的区别是什么？**

* **答案：** ArrayList 基于动态数组实现，支持随机访问，查询快（$O(1)$），增删慢；LinkedList 基于双向链表实现，不支持随机访问，查询慢，但增删快（$O(1)$）。

#### **23\. Java 中 \== 和 equals() 的区别？**

* **答案：** \== 对于基本类型比较的是值，对于引用类型比较的是内存地址；equals() 是 Object 的方法，默认也是比较地址，但很多类（如 String）重写了该方法用于比较对象的内容是否相等。

#### **24\. 为什么重写 equals() 时必须重写 hashCode()？**

* **答案：** 维护二者的通用约定：如果两个对象通过 equals() 比较相等，那么它们的 hashCode() 必须返回相同的值。否则在诸如 HashMap、HashSet 等基于哈希表的集合中会出现数据错乱和找不到对象的情况。

#### **25\. 什么是内存泄漏（Memory Leak）？**

* **答案：** 内存泄漏是指程序中已经不再使用的对象由于被其他长期存活的对象持有引用，导致垃圾回收器无法将其从内存中回收，长此以往会导致内存耗尽发生 OOM。

## **第二部分：Android 核心组件与机制（第 26-55 题）**

#### **26\. Android 的四大组件是什么？**

* **答案：** Activity（界面展示）、Service（后台服务）、BroadcastReceiver（广播接收器）、ContentProvider（数据共享）。

#### **27\. 请描述 Activity 的正常生命周期流程。**

* **答案：** onCreate() \-\> onStart() \-\> onResume() \-\> \[运行状态\] \-\> onPause() \-\> onStop() \-\> onDestroy()。

#### **28\. 当 Activity A 启动 Activity B 时，两者的生命周期方法执行顺序是怎样的？**

* **答案：** A.onPause() \-\> B.onCreate() \-\> B.onStart() \-\> B.onResume() \-\> A.onStop()。

#### **29\. Activity 的四种启动模式（LaunchMode）分别是什么？**

* **答案：** standard（标准模式）、singleTop（栈顶复用模式）、singleTask（栈内复用模式）、singleInstance（单例模式）。

#### **30\. singleTask 启动模式有什么特点？**

* **答案：** 如果全局或指定任务栈内已经存在该 Activity 的实例，则会将该实例之上的所有其他 Activity 清出栈（Clear Top），使其露到栈顶并回调 onNewIntent()，而不会重新创建实例。

#### **31\. 什么是横竖屏切换时 Activity 的生命周期变化？如何阻止其重新创建？**

* **答案：** 默认情况下横竖屏切换会导致 Activity 销毁并重建（执行 onDestroy 随后 onCreate）。可以在 AndroidManifest 中为该 Activity 配置 android:configChanges="orientation|screenSize"，这样切换时不会重建，而是回调 onConfigurationChanged() 方法。

#### **32\. Fragment 的生命周期与 Activity 有什么关系？**

* **答案：** Fragment 必须依附于 Activity 存在。其生命周期受宿主 Activity 的直接影响。Fragment 独有的生命周期方法包括 onAttach()、onCreateView()、onActivityCreated()、onDestroyView()、onDetach() 等。

#### **33\. Service 有哪两种启动方式？它们有什么区别？**

* **答案：** startService() 和 bindService()。startService 启动的服务与调用者没有绑定关系，调用者销毁后服务依然在后台运行；bindService 启动的服务与调用者绑定，调用者销毁时服务也会随之解绑并销毁。

#### **34\. Service 的 onStartCommand() 返回值有哪些？各代表什么含义？**

* **答案：** START\_STICKY（被系统杀死后尝试重建服务，但不保留 Intent）；START\_NOT\_STICKY（被杀死后不重建）；START\_REDELIVER\_INTENT（被杀死后重建并重新投递最后的 Intent）。

#### **35\. 什么是 IntentService？它有什么优势？**

* **答案：** IntentService 是 Service 的子类。它内部自带了一个工作线程（HandlerThread），可以用来处理异步请求，当所有请求处理完毕后，它会自动停止服务，不需要手动调用 stopSelf()。

#### **36\. 什么是 BroadcastReceiver？如何注册广播？**

* **答案：** 广播接收器是用于接收来自系统或其他应用发出的广播通知的组件。注册方式有两种：静态注册（在 AndroidManifest.xml 中声明，常驻型）和动态注册（在代码中通过 registerReceiver() 注册，随生命周期注销）。

#### **37\. 什么是本地广播（LocalBroadcastManager）？它有什么好处？**

* **答案：** 本地广播是指发出的广播只在应用内部传播。好处是安全性高（外部应用无法接收，也无法向应用内发送伪造广播），且比全局广播更高效。

#### **38\. ContentProvider 的核心作用是什么？**

* **答案：** 跨应用数据共享。它为不同的应用之间互相访问和操作数据（如读取系统通讯录）提供了一套标准的安全接口。

#### **39\. 简述 Android 中的 Handler 机制组成。**

* **答案：** 核心由四部分组成：Handler（发送和处理消息）、Message（消息载体）、MessageQueue（消息队列）、Looper（消息循环器，负责从队列中轮询抽取消息）。

#### **40\. 为什么 Android 不允许在子线程中更新 UI？**

* **答案：** 因为 Android 的 UI 控件不是线程安全的。如果多线程并发修改 UI 可能会导致界面状态错乱。引入 Handler 机制可以将所有 UI 修改操作收敛并同步到主线程执行。

#### **41\. 一个线程可以有几个 Looper？几个 MessageQueue？几个 Handler？**

* **答案：** 一个线程只能有一个 Looper 和一个 MessageQueue；但一个线程可以创建无数个 Handler，它们都共享同一个 Looper。

#### **42\. Handler 导致的内存泄漏原因是什么？如何解决？**

* **答案：** 非静态内部类或匿名内部类的 Handler 会隐式持有外部类（Activity）的强引用。如果主线程消息队列中还有未处理完的消息，消息持有 Handler，Handler 持有 Activity，导致 Activity 退出时无法被 GC 回收。解决方法是：将 Handler 定义为静态内部类，内部通过 WeakReference（弱引用）持有 Activity。

#### **43\. 什么是子线程中的 Looper.prepare() 和 Looper.loop()？**

* **答案：** 主线程（UI 线程）默认已经帮我们初始化好了 Looper。而自定义的普通子线程默认是没有 Looper 的，若想在子线程中使用 Handler 接收消息，必须先调用 Looper.prepare() 创建队列，再调用 Looper.loop() 开启循环。

#### **44\. 什么是 View 的事件分发机制？涉及哪些核心方法？**

* **答案：** 事件分发是指当一个点击事件（MotionEvent）产生后，系统将该事件传递给具体的 View 进行响应的过程。核心方法有三个：dispatchTouchEvent()（分发事件）、onInterceptTouchEvent()（拦截事件，仅 ViewGroup 有）、onTouchEvent()（消费/处理事件）。

#### **45\. 如果子 View 的 onTouchEvent() 返回了 false，事件会如何传递？**

* **答案：** 事件会回传给其父容器（ViewGroup）的 onTouchEvent() 处理，如果一直没有人消费，最终事件会交由 Activity 的 onTouchEvent() 处理。

#### **46\. 简述 View 的绘制流程。**

* **答案：** 绘制分为三个主要阶段：measure（测量，确定 View 的大小）、layout（布局，确定 View 在父容器中的摆放位置）、draw（绘制，将 View 的画面画到屏幕 Canvas 上）。

#### **47\. View.invalidate() 和 View.requestLayout() 的区别？**

* **答案：** invalidate() 会触发 View 的重新绘制（只会调用 draw() 流程），适用于刷新控件内容；requestLayout() 会触发重新测量、布局和绘制（完整走一遍 measure \-\> layout \-\> draw），适用于控件大小或位置发生变化时。

#### **48\. 什么是 Context？它的作用是什么？**

* **答案：** Context 是一个抽象类，代表当前应用程序环境的上下文。通过它可以访问应用级别的资源（如 asset、res）、启动组件（Activity、Service）、发送广播等。

#### **49\. Application Context 和 Activity Context 有什么区别？**

* **答案：** Application Context 生命周期贯穿整个应用的生命周期，适合做全局初始化；Activity Context 生命周期与当前 Activity 绑定，用于跟 UI 相关的操作（如弹出 Dialog、启动 Activity、动态加载布局）。

#### **50\. 什么是 AsyncTask？为什么它现在被废弃了？**

* **答案：** AsyncTask 是 Android 提供的一个轻量级异步任务类。废弃原因主要是容易造成内存泄漏、生命周期未与 Activity 绑定、多个任务并发执行时的线程排队管理复杂等。

#### **51\. 什么是 Android 序列化？Serializable 和 Parcelable 的区别？**

* **答案：** 将对象转换为字节流以便传输或存储的过程。Serializable 是 Java 提供的标准接口，使用反射，性能较差但使用简单；Parcelable 是 Android 专有接口，通过将对象内存打包实现，效率极高，推荐在 Android 组件间（如 Intent 传值）使用。

#### **52\. 什么是 ANR？什么情况下会触发 ANR？**

* **答案：** ANR（Application Not Responding）即应用无响应。通常在主线程执行了耗时操作时触发。标准为：Activity 响应输入事件（如点击）超时 5 秒；BroadcastReceiver 执行超时 10 秒；Service 执行超时 20 秒。

#### **53\. 如何定位和分析 ANR？**

* **答案：** 查看系统生成的 traces.txt 文件（或通过 data/anr/ 目录下的日志），寻找主线程（main）的堆栈信息，分析是哪个方法阻塞了主线程；平时开发中应避免在主线程做网络请求或复杂的 I/O 操作。

#### **54\. 什么是 OOM？Android 中如何避免 OOM？**

* **答案：** OOM（Out Of Memory）即内存溢出。避免方法包括：及时释放不需要的资源；使用轻量级数据结构；大图加载使用采样率压缩（BitmapFactory.Options）；避免长生命周期对象持有短生命周期对象的引用（防止内存泄漏）。

#### **55\. 什么是卡顿（Jank）？Android 每秒刷新多少帧算流畅？**

* **答案：** 卡顿是指由于主线程耗时导致错过屏幕刷新信号。Android 标准的屏幕刷新率通常是 60Hz，即每 16.67 毫秒刷新一帧，如果一帧的计算和渲染超过这个时间，就会产生掉帧和卡顿。

## **第三部分：高级进阶、并发与架构（第 56-85 题）**

#### **56\. 什么是多进程？Android 中如何让组件运行在独立进程中？**

* **答案：** 多进程指一个应用中包含多个运行在独立虚拟机中的进程。可以在 AndroidManifest 文件的组件标签中配置 android:process=":remote" 属性。

#### **57\. 跨进程通信（IPC）有哪些常见方式？Android 最核心的是什么？**

* **答案：** 常见方式包括：Bundle、文件共享、Messenger、AIDL、ContentProvider、Socket。最核心、最底层的是基于 Binder 机制。

#### **58\. 为什么 Android 选择 Binder 作为核心 IPC 机制？**

* **答案：** \* 性能好：数据只需拷贝一次（通过内存映射 mmap）。  
  * 安全性高：Binder 协议支持在内核层面对调用者的 UID/PID 进行安全检查。  
  * 稳定易用：基于 C/S 架构，使用面向对象的思想封装。

#### **59\. 什么是 AIDL？其全称是什么？**

* **答案：** AIDL（Android Interface Definition Language）即 Android 接口定义语言。它用于定义客户端和服务端通行的接口契约，编译器会自动根据它生成跨进程通信的具体桩代码（Stub）。

#### **60\. 什么是线程池？为什么要使用线程池？**

* **答案：** 线程池是一种管理和复用线程的资源池。使用它可以避免频繁创建和销毁线程带来的系统开销，控制最大并发线程数，提高系统稳定性。

#### **61\. Java 线程池的核心参数有哪些？**

* **答案：** corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（空闲线程存活时间）、unit（时间单位）、workQueue（任务等待队列）、threadFactory（线程工厂）、handler（拒绝策略）。

#### **62\. 什么是死锁？产生死锁的四个必要条件是什么？**

* **答案：** 死锁指两个或多个线程在执行过程中，因争夺资源而造成的一种互相等待的僵局。必要条件：互斥条件、不可剥夺条件、请求与保持条件、循环等待条件。

#### **63\. synchronized 和 ReentrantLock 的区别是什么？**

* **答案：** synchronized 是 Java 关键字，由 JVM 层面实现，自动加锁和释放锁；ReentrantLock 是 JDK 提供的 API 类，需要手动加锁和手动在 finally 中释放锁，且支持中断响应、公平锁和超时获取。

#### **64\. 什么是 volatile 关键字？**

* **答案：** 用于修饰变量。它能保证该变量对所有线程的可见性（一个线程修改了值，其他线程立刻能看到最新值），并且禁止指令重排序，但不保证原子性。

#### **65\. 什么是 CAS（Compare And Swap）？**

* **答案：** 无锁优化（自旋锁）的核心思想。它包含三个参数：内存位置 V、预期原值 A 和新值 B。当且仅当内存位置的值等于 A 时，才将值更新为 B，否则不执行任何操作并重试，从而实现多线程安全的原子操作。

#### **66\. 什么是多级缓存架构（如图片加载框架、LRU 缓存）？**

* **答案：** 通常分为三级：内存缓存（读取速度最快）、本地磁盘缓存（容量大，持久化存储）、网络源（最慢）。依次向下查找，找到后逆向回填。

#### **67\. 请简述 LRU 算法的原理。**

* **答案：** LRU（Least Recently Used）即最近最少使用算法。其核心思想是如果数据最近被访问过，那么将来被访问的几率也更高。通常采用“哈希表+双向链表”实现，当缓存达到上限时，淘汰链表头部（最久未被访问）的数据。

#### **68\. Android 中常见的应用架构有哪些？**

* **答案：** MVC（Model-View-Controller）、MVP（Model-View-Presenter）、MVVM（Model-View-ViewModel）。

#### **69\. MVP 架构中 Presenter 的主要职责是什么？**

* **答案：** Presenter 是 View 和 Model 之间的桥梁。它负责从 Model 获取数据，处理业务逻辑，并通过接口回调控制 View 的界面刷新，从而实现 View 和 Model 的解耦。

#### **70\. MVVM 架构相比 MVP 有什么优势？**

* **答案：** MVVM 引入了数据驱动（Data Binding / LiveData）的概念。ViewModel 内部持有可观察的数据，当数据改变时 UI 会自动更新，减少了 MVP 中大量繁琐的接口契约代码。

#### **71\. 什么是 Jetpack？包含哪些核心组件？**

* **答案：** Jetpack 是谷歌推出的一套官方库、工具和架构指南。核心组件包括：Lifecycle（生命周期感知）、LiveData（响应式数据持有）、ViewModel（存放 UI 数据）、Room（数据库管理）等。

#### **72\. LiveData 为什么能避免内存泄漏？**

* **答案：** 因为 LiveData 能够感知 LifecycleOwner（如 Activity/Fragment）的生命周期。当宿主生命周期处于 DESTROYED 状态时，它会自动移除观察者，防止产生持有强引用导致的泄漏。

#### **73\. 什么是热修复（HotFix）技术？**

* **答案：** 在不重新发布应用新版本的情况下，通过在后台下发补丁包（Dex/Resource/So 等），动态修复线上应用 Bug 的技术。

#### **74\. 热修复中 ClassLoader 类的加载方案原理是什么？**

* **答案：** Android 的 BaseDexClassLoader 内部持有一个 DexPathList，它包含一个 Element\[\] dexElements 数组。当类加载器查找类时会遍历该数组。热修复就是将修复好 Bug 的补丁 Dex 插入到该数组的最前面，使其先被找到并加载，达到“封杀”旧 Bug 类的效果。

#### **75\. 什么是插件化？它和热修复有什么区别？**

* **答案：** 插件化是将应用拆分为多个独立的模块（插件 Dex），在运行时动态加载未安装的 apk/dex 并运行。区别在于：热修复侧重于修复既有 Bug，而插件化侧重于功能的动态扩展和按需下载。

#### **76\. 什么是组件化？**

* **答案：** 组件化是一种架构设计思想。它在编码期将大项目按照业务功能拆分为多个独立的 Module（组件），每个组件可以单独配置、独立编译运行，也可以拼装在一起形成最终的完整 App，解决团队开发代码耦合严重的问题。

#### **77\. 什么是反射（Reflection）？在 Android 中有什么应用和代价？**

* **答案：** 反射是在运行时动态获取类的信息、属性、方法并进行调用的机制。常用于访问系统隐藏的 API（@hide）或框架解耦。代价是会产生较大的性能开销，且破坏了类的封装性。

#### **78\. 什么是 ProGuard（混淆）？**

* **答案：** 在编译时通过缩短类名、方法名（变成 a, b, c），移除未使用的代码和属性，来减小 APK 体积并增加反编译难度、保护代码资产的工具。

#### **79\. 动态布局（XML）和自绘 UI（Canvas 绘图）的区别？**

* **答案：** XML 布局通过系统解析器生成 View 树，由系统测量和摆放，简单直观；自绘 UI 则直接在 Canvas 上调用 drawXXX 绘制像素，可以完全绕过原生控件体系，实现高度定制的高性能复杂动效。

#### **80\. 什么是屏幕适配？有哪些常用适配方案？**

* **答案：** 确保应用在不同分辨率和尺寸的屏幕上显示效果一致。方案包括：dp/sp 基础单位适配、百分比布局、多套最小宽度限定符（SmallestWidth 适配）、修改系统 Density 的今日头条适配方案。

#### **81\. 什么是内存抖动（Memory Churn）？**

* **答案：** 在极短的时间内频繁地创建和销毁大量对象（例如在 onDraw() 方法或循环里 new 对象），导致垃圾回收器（GC）频繁启动，占用主线程执行时间，从而引发界面卡顿。

#### **82\. 什么是 App 启动优化？分为哪几种启动方式？**

* **答案：** 优化用户从点击桌面图标到看到界面的耗时。分为冷启动（进程不存在，需建进程和初始化 Application）、温启动（进程存在但 Activity 被销毁）、热启动（进程和 Activity 都存在，直接移到前台）。

#### **83\. 如何优化应用的冷启动时间？**

* **答案：** 减少 Application.onCreate() 中的第三方库同步初始化操作（改为异步加载或延迟加载）；优化第一个 Activity 的布局结构；配置默认启动主题背景（消除白屏/黑屏）。

#### **84\. APK 的瘦身（优化体积）有哪些手段？**

* **答案：** 开启混淆和代码裁剪；图片转为 WebP 格式；使用 VectorDrawable（矢量图）；通过 ndk.abiFilters 移除不需要的 So 库架构；利用 Lint 移除未使用的资源。

#### **85\. 网络请求优化有哪些常见策略？**

* **答案：** 减少请求次数（合并接口）；启用 Http 缓存机制；数据压缩（使用 Protocol Buffers 代替 JSON）；合理设置超时时间与重试机制。

## **第四部分：设计模式与现代 UI 框架对比（第 86-100 题）**

#### **86\. 面向对象编程（OOP）中，继承、接口和装饰器这三者的核心区别是什么？**

* **答案：** 它们用于解决代码复用和扩展的不同维度问题：  
  * **继承：** 决定你“是什么”（Is-A 关系），属于出生血统和强耦合复用。  
  * **接口：** 决定你“能做什么”（Like-A 关系），是一种解耦的能力契约。  
  * **装饰器：** 决定你“怎么精装/变强”（Has-A 关系），在不改变原类的基础上动态添加Buff/新功能。

#### **87\. 继承（Inheritance）的最大优缺点是什么？**

* **答案：** 优点是代码高度复用，子类自动获取父类方法；缺点是强耦合（高代系依赖），父类改变子类可能崩溃，且大多数主流语言只支持单继承。

#### **88\. 为什么在开发中提倡“多用组合，少用继承”？**

* **答案：** 组合（将一个对象作为另一个类的属性）可以在运行时动态改变行为，是低耦合的；而继承在编译期就固定了类关系，是强耦合的，过度使用会导致继承树极度臃肿、缺乏弹性。

#### **89\. 接口（Interface）的主要目的是什么？**

* **答案：** 主要目的是**规范化和解耦**。调用者只需关注接口定义的行为契约，不关心底层的具体类是什么，实现面向接口编程。

#### **90\. 什么是装饰器模式（Decorator Pattern）？它有什么优势？**

* **答案：** 装饰器模式允许向一个现有的对象添加新的功能，同时又不改变其结构。优势是比生成子类更加灵活，可以组合不同的装饰器动态扩展对象功能，且不污染原有类的代码。

#### **91\. 试举一个 Java 原生类库中使用装饰器模式的例子。**

* **答案：** Java 的 I/O 流体系。例如 FileInputStream 是标准读取流，通过包装成 BufferedInputStream(new FileInputStream(file))，动态为原流添加了“高效缓冲区”的功能，而两者的基类都是 InputStream。

#### **92\. 单例模式的懒汉式“双重检查锁定（DCL）”为什么需要加 volatile 关键字？**

* **答案：** 防止编译器进行指令重排序。因为 instance \= new Singleton() 在底层包含分配内存、初始化对象、将引用指向内存三个步骤，如果不加 volatile，可能会出现步骤颠倒，导致另一个线程拿到一个未初始化完成的半成品对象。

#### **93\. 什么是观察者模式（Observer Pattern）？Android 中有什么代表？**

* **答案：** 定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都会得到通知并自动更新。Android 中的代表有：RxJava、LiveData、各种 Listener 监听机制。

#### **94\. 什么是工厂模式（Factory Pattern）？**

* **答案：** 隐藏创建对象的具体逻辑，将对象的实例化过程封装在工厂方法中，客户端通过统一的工厂接口或方法传入参数来获取对象，降低了代码的耦合度。

#### **95\. 什么是策略模式（Strategy Pattern）？**

* **答案：** 定义一系列算法，把它们一个个封装起来，并且使它们可以相互替换。使得算法可以独立于使用它的客户端而变化（如 Android 中为图片加载框架设计统一的抽象接口，底层可随意切换 Glide 或 COIL）。

#### **96\. 目前主流三大 UI 框架 Flutter、Jetpack Compose、React 的根本区别体现在哪里？**

* **答案：** 体现在底层的语言机制、渲染引擎和内存模型不同：  
  * **Flutter：** 使用 Dart 语言（AOT 编译），完全自绘像素，绕过原生系统平台，直接通过 Skia/Impeller 绘制在屏幕上。  
  * **Jetpack Compose：** 基于 Kotlin，属于半自绘/半原生，利用编译器插件状态机，在 Android 上直接向硬件加速 Canvas 绘图。  
  * **React：** 基于 JavaScript/TypeScript（JIT 执行），通过浏览器内核或宿主环境的原生 DOM 引擎去布局和绘制。

#### **97\. Flutter 框架最大的性能瓶颈和优势分别是什么？**

* **答案：** 优势是全平台像素级UI高度一致，渲染极其流畅；瓶颈是引擎包体积较大、初始化引擎加载耗时，以及 Dart 与原生平台进行跨端通信（MethodChannel）时存在桥接开销。

#### **98\. Jetpack Compose 框架的运行时内存模型和更新触发机制是什么？**

* **答案：** 内存模型是极致扁平的线性数据结构（Slot Table）+ 局部轻量的 LayoutNode 树；更新机制是通过快照系统（Snapshot）触发局部函数重组（Recomposition）。

#### **99\. Jetpack Compose 中所谓的“智能跳过（Smart Recomposition）”是指什么？**

* **答案：** 编译器在编译期会识别函数参数的稳定性（Stable）。如果输入参数在下一次重组时没有发生变化，Compose 会直接跳过该函数的执行并返回上一次的结果，从而大幅提升更新性能。

#### **100\. 简述 React 框架的核心优化手段与机制。**

* **答案：** 采用双缓存 Fiber 链表树进行并发调度。核心优化手段是通过并发调度器（Scheduler）进行异步任务切片与时间分片，结合虚拟 DOM 的 Diff 算法及 key 属性，实现对宿主环境最小范围的 DOM 操作更新。
