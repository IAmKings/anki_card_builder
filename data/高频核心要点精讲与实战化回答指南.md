# **Java 基础篇**

## **一、 多线程与线程池**

* **多线程开关控制：** 常用 volatile 修饰布尔标记位实现温和退出；或者使用 Thread.interrupt() 中断信号，并在循环中通过 isInterrupted() 检查。  
* **并发访问：** \* **锁机制：** synchronized（内置锁，可重入）和 ReentrantLock（显式锁，支持公平锁、中断响应、超时获取）。  
  * **无锁方案：** AtomicInteger 等 CAS 原子类，ThreadLocal（线程隔离）。  
* **线程池的作用：** 1\. **降低资源消耗**：重用已创建的线程。  
  2\. **提高响应速度**：任务到达时无需等待线程创建。  
  3\. **提高线程的可管理性**：避免无限制创建线程导致内存溢出（OOM），可控制并发数。

## **二、 HTTP 编程**

* **GET 与 POST 的区别：**

| 特性 | GET | POST |
| :---- | :---- | :---- |
| **数据位置** | 拼接到 URL 后面（Query String） | 放在请求体（Request Body）中 |
| **安全性** | 较低（参数对所有人可见、留有历史记录） | 相对较高（但仍需 HTTPS 加密） |
| **数据长度** | 受浏览器/服务器 URL 长度限制（通常 2KB 左右） | 原则上无限制 |
| **幂等性** | 幂等（多次请求结果相同，只读） | 非幂等（用于创建/修改数据） |

* **大文件上传：** 核心是**分片上传（Chunked）与断点续传**。将大文件切分成 N 个固定大小的 Block，每次上传一个 Block，服务器记录已接收的切片索引。若中途网络断开，下次从失败的切片处继续上传。

## **三、 集合框架（ArrayList, LinkedList 等）**

* **ArrayList vs Vector vs LinkedList：**  
  * ArrayList：基于**动态数组**。随机访问快（$O(1)$），尾部增删快，中间增删慢（需要移动元素）。线程不安全。  
  * Vector：实现与 ArrayList 类似，但所有方法加了 synchronized，**线程安全**，性能低。  
  * LinkedList：基于**双向链表**。增删快（只需修改指针，$O(1)$），随机访问慢（需要遍历，$O(n)$）。  
* **HashMap vs Hashtable：**  
  * HashMap：线程不安全，允许 key/value 为 null。底层是数组+链表+红黑树（JDK 8+）。  
  * Hashtable：线程安全（方法加锁），不允许 key/value 为 null，老旧不推荐，高并发改用 ConcurrentHashMap。

# **Android 基础篇**

## **一、 四大组件及作用**

1. **Activity（活动）：** 应用程序的界面展示与用户交互入口。  
2. **Service（服务）：** 无界面，在后台执行长生命周期操作（如音乐播放、大文件下载）。  
3. **BroadcastReceiver（广播接收者）：** 用于接收来自系统或其他应用发出的广播通知（如低电量、网络变化），实现应用内或应用间解耦。  
4. **ContentProvider（内容提供者）：** 管理应用的数据，并提供接口供其他应用安全地进行跨进程数据访问（如读取系统通讯录）。

## **二、 Activity 启动模式**

* **Standard（标准模式）：** 默认。每次启动都会创建一个新的实例压入启动它的那个栈顶。  
* **SingleTop（栈顶复用）：** 如果新 Activity 已经位于任务栈的**栈顶**，则不创建新实例，而是复用它并回调 onNewIntent()。*（适用场景：消息推送通知栏点击打开聊天界面）*  
* **SingleTask（栈内复用）：** 如果目标 Activity 在栈中存在，则将其上方的所有 Activity 清出栈（Clear-Top），使其处于栈顶并回调 onNewIntent()。*（适用场景：应用主页/MainActivity）*  
* **SingleInstance（单实例模式）：** 独占一个全新的任务栈，整个系统内只有一个实例。*（适用场景：来电通话界面、闹钟提醒界面）*

## **三、 内存泄露（Memory Leak）常见场景**

**本质：** 长生命周期的对象持有短生命周期对象的强引用，导致短生命周期对象无法被 GC 回收。

1. **非静态内部类/匿名内部类：** 如 Handler 或 AsyncTask 持有 Activity 的隐式引用，若在 Activity 销毁时后台任务未结束，导致 Activity 泄露。  
   * *解法：改为静态内部类 \+ 弱引用（WeakReference）。*  
2. **单例模式：** 单例的生命周期与 Application 一样长，如果单例内部传入并持有了 Activity 的 Context。  
   * *解法：尽量使用 context.getApplicationContext()。*  
3. **静态变量：** 静态 View 或静态 Activity 变量未及时置空。  
4. **资源未关闭：** Cursor、IO 流、Bitmap（早期版本）、BraodcastReceiver 未注销、动画未停止。

# **Android 提升篇**

## **一、 数据库模块（SQLite3）**

* **批量插入优化：** 必须使用**显示事务（Transaction）**。  
  Java  
  db.beginTransaction();  
  try {  
      for (Data data : list) {  
          db.insert(..., contentValues);  
      }  
      db.setTransactionSuccessful(); // 标记成功  
  } finally {  
      db.endTransaction(); // 真正一次性提交  
  }

  *原理：不使用事务时，SQLite 默认会将每条 insert 语句当成独立事务，频繁读写磁盘；使用事务后，多条数据在内存中处理，最后一次性写入磁盘。*  
* **多线程并发访问：** SQLite 默认不支持多个线程同时写入。会报 database is locked 错误。  
  * *做法：* 确保整个应用中只使用**同一个 SQLiteOpenHelper 单例**。利用其内部的同步锁机制，或结合 Java 的 ReentrantLock 进行读写分离控制。

## **二、 Activity 启动过程 (startActivity 简析)**

1. **发起方：** Activity.startActivity() \-\> 通过 Instrumentation 向 **AMS (ActivityManagerService)** 发起跨进程 IPC 请求。  
2. **系统服务：** AMS 接收请求，检查目标 Activity 的清单配置、启动权限以及启动模式，处理任务栈。  
3. **进程创建：** 如果目标 Activity 所在的应用进程未启动，AMS 通过 Zygote 进程 fork 出一个新的应用进程。  
4. **应用绑定：** 新进程启动后，其 ActivityThread 的 main() 方法执行，向 AMS 绑定 Application。  
5. **生命周期回调：** AMS 通过 Binder 驱动通知应用进程的 ApplicationThread，最终通过 H (Handler) 切换到主线程，由 Instrumentation 反射创建 Activity 实例并执行 onCreate() \-\> onStart() \-\> onResume()。

## **三、 事件分发机制**

Android 的事件分发主要由三大方法协作完成：dispatchTouchEvent()、onInterceptTouchEvent()、onTouchEvent()。

* **分发流程（U型走势）：**  
  1. **Activity** 接收到触摸事件，调用 dispatchTouchEvent()。  
  2. 向下传递给 **ViewGroup**，ViewGroup 的 dispatchTouchEvent() 触发。  
  3. ViewGroup 会调用 onInterceptTouchEvent() 询问是否拦截。  
     * 若**拦截**（返回 true），事件不再向下传递，交给自身的 onTouchEvent() 处理。  
     * 若**不拦截**（返回 false），事件继续向下传递给子 View。  
  4. 到了底层的 **View**，没有 onInterceptTouchEvent()，直接调用 onTouchEvent()。  
  5. 如果消费了（返回 true），事件结束；如果不消费（返回 false），事件开始**逆向向上回溯**，传回父布局的 onTouchEvent()，直到 Activity。

# **Android 系统篇**

## **一、 Dalvik 与 ART 的理解**

* **Dalvik VM：** \* 基于架构：**基于寄存器**（相比 JVM 基于栈，运行速度更快，指令更紧凑）。  
  * 编译机制：**JIT (Just-In-Time) 即时编译**。应用运行时，将字节码实时翻译成机器码，每次运行都要翻译，导致App运行效率相对低、耗电。  
* **ART (Android Runtime)：** 升级替代品。  
  * 编译机制：**AOT (Ahead-Of-Time) 预编译**。在应用安装时就全量翻译成机器码，App 运行速度大幅提升（空间换时间）。  
  * 现代优化（Android 7.0+）：采用 **混合编译（AOT \+ JIT \+ 配置文件引导）**。安装时不预编译，运行时 JIT，并在设备空闲且充电时将常用代码 AOT 编译，完美平衡了安装速度、存储空间和运行性能。

## **二、 Android 开机启动过程**

1. **Power On (电源启动)：** 引导芯片代码从 ROM 开始执行，加载 **Bootloader** 到 RAM。  
2. **Bootloader (引导程序)：** 初始化硬件参数，拉起 **Linux 内核**。  
3. **Kernel (内核层)：** 初始化设置、驱动程序。加载完毕后，在用户空间启动第一个进程 —— **init 进程**。  
4. **Init 进程：** 解析 init.rc 配置文件，启动守护进程。最关键的是启动了 **Zygote (孵化器进程)**。  
5. **Zygote 进程：** 初始化 Java 运行环境（创建虚拟机，预加载系统的类和资源）。然后通过 fork 启动 **SystemServer 进程**。  
6. **SystemServer 进程：** 启动 Android 核心系统服务（AMS, WMS, PMS, ServiceManager 等）。  
7. **Launcher (桌面)：** 当所有系统服务准备就绪，AMS 会启动系统第一个 Activity —— Launcher，用户即可看到桌面。
