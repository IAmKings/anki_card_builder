# **首席 Android 架构师/面试官视角：高频核心面试题集（精选第一部分）**

## **第一部分：Java 基础、JVM 与并发编程（第 1-30 题）**

### **1\. 简述 Java 中继承、接口和装饰器模式的区别，你在项目中如何选择？**

* **答案：** \* **继承（Inheritance）**：建立的是 “Is-A”（是一个）的关系。核心目的是**代码复用**。子类强耦合于父类，Java 仅支持单继承。  
  * **接口（Interface）**：建立的是 “Like-A”（具备某种能力）的关系。它是一种**行为契约和规范**，核心目的是解耦，实现多态。  
  * **装饰器模式（Decorator）**：建立的是 “Has-A”（组合）的关系。在不改变原有类结构和继承体系的情况下，**动态地为对象扩展新功能**（增强/换肤），比继承更具弹性。  
  * **项目选择**：若有明确的血统延续和通用基础逻辑复用，用继承；若要定义模块间的通信标准或解耦复杂业务，用接口；若需要动态叠加多种行为且避免类爆炸，用装饰器模式（如 Android 中的 ContextWrapper）。

### **2\. JVM 的运行时数据区包含哪些部分？哪些是线程私有的，哪些是线程共享的？**

* **答案：** \* 包含五大部分：**程序计数器**、**Java虚拟机栈**、**本地方法栈**、**Java堆**、**方法区**（以及其中的运行时常量池）。  
  * **线程私有**：程序计数器、Java 虚拟机栈、本地方法栈。它们随线程创建而创建，随线程销毁而销毁。  
  * **线程共享**：Java 堆、方法区。所有线程均可访问，是垃圾回收（GC）的主战场。

### **3\. 什么是 JVM 的“双亲委派机制”？它有什么好处？在 Android 中有什么变形？**

* **答案：** \* **机制**：当一个类加载器收到类加载请求时，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成，只有当父类加载器反馈自己无法完成这个加载请求时，子加载器才会尝试自己去加载。  
  * **好处**：保证 Java 核心 API 的安全和唯一性，防止核心类（如 java.lang.String）被恶意篡改。  
  * **Android 中的变形**：Android 采用的是 PathClassLoader 和 DexClassLoader（继承自 BaseDexClassLoader）。虽然也遵循双亲委派，但其加载的是 .dex 文件而非 .class ；另外在一些热修复和插件化框架中，会通过 Hook 或重写 loadClass 破坏双亲委派，从而优先加载修复后的本地类。

### **4\. 强引用、软引用、弱引用、虚引用的区别是什么？Android 中常见应用场景有哪些？**

* **答案：** \* **强引用（StrongReference）**：最普遍的引用。只要强引用存在，垃圾回收器永远不会回收掉被引用的对象，宁愿抛出 OOM。  
  * **软引用（SoftReference）**：发生内存溢出（OOM）之前，系统会把这些对象列进回收范围进行第二次回收。如果这次回收还没有足够的内存，才会抛出 OOM。**场景**：适合做图片或数据的缓存。  
  * **弱引用（WeakReference）**：只要垃圾回收器开始工作，无论当前内存是否足够，都会回收掉只被弱引用关联的对象。**场景**：解决内存泄漏，如 WeakReference\<Activity\> 用于 Handler 或异步任务中。  
  * **虚引用（PhantomReference）**：形同虚设，无法通过它获取对象实例。其唯一目的就是在这个对象被回收时收到一个系统通知。**场景**：跟踪对象被销毁的活动，进行堆外内存释放。

### **5\. String、StringBuffer 和 StringBuilder 的区别是什么？**

* **答案：** \* **String**：不可变字符序列，底层是 final char\[\]（或 byte\[\]），任何修改都会生成新的 String 对象，线程安全。  
  * **StringBuffer**：可变字符序列，对方法加了 synchronized 锁，**线程安全**，效率较低。  
  * **StringBuilder**：可变字符序列，没有加锁，**线程不安全**，但在单线程下效率最高。

### **6\. Java 中 equals() 和 hashCode() 的关系是什么？为什么重写前者必须重写后者？**

* **答案：** \* 依据 Java 规范：若两个对象通过 equals() 比较相等，那么它们的 hashCode() 必须相同；反之，hashCode() 相同，对象不一定相等（哈希冲突）。  
  * **原因**：在 HashMap、HashSet 等基于哈希表实现的集合中，判断元素重复的流程是先计算 hashCode 定位桶位置，若哈希值相同，再调用 equals() 比较内容。如果重写了 equals 而不重写 hashCode，会导致两个内容完全相同的对象计算出不同的哈希值，从而同时存入集合，破坏了集合的唯一性。

### **7\. 谈谈你对 volatile 关键字的理解，它能保证原子性吗？**

* **答案：** \* volatile 是 Java 虚拟机提供的最轻量级的同步机制，具备两大核心特性：  
  1\. **保证可见性**：当一个线程修改了共享变量的值，新值对于其他线程来说是立即可以得知的（强制跳过工作内存，直接读写主内存）。  
  2\. **禁止指令重排序优化**：通过加入内存屏障，阻止编译器和处理器为了优化性能而打乱代码执行顺序。  
  * **原子性**：它**不能**保证原子性。对于复合操作（如 i++），volatile 无法阻止多线程交叉执行，仍需使用 synchronized 或 AtomicInteger。

### **8\. 为什么在单例模式中，双重检查锁（DCL）模式必须配合 volatile 关键字？**

* **答案：** \* 主要是为了防止**指令重排序**导致的“半初始化对象”问题。  
  * instance \= new Singleton(); 在底层并非原子操作，而是分为三步：  
    1. 分配对象的内存空间；  
    2. 初始化对象；  
    3. 将 instance 引用指向刚分配的内存。  
  * 如果没有 volatile，编译器可能将顺序重排为 1 \-\> 3 \-\> 2。当线程 A 执行完第 3 步但尚未初始化完成（第 2 步）时，线程 B 恰好执行到外层判空，此时判断 instance \!= null 成立，直接返回了该半成品对象，导致线程 B 使用时发生异常。

### **9\. 简述 synchronized 与 ReentrantLock 的区别。**

* **答案：** \* **实现层面**：synchronized 是 Java 关键字，依赖于 JVM 级别的 Monitor 锁来实现（属于内置锁）；ReentrantLock 是 JDK 提供的 API，基于 AQS（AbstractQueuedSynchronizer）实现。  
  * **锁的释放**：synchronized 在代码块结束或异常时由 JVM 自动释放锁；ReentrantLock 必须在 finally 块中手动调用 unlock() 释放，否则易造成死锁。  
  * **高级特性**：ReentrantLock 支持更多高级功能：可以指定是公平锁还是非公平锁（synchronized 只能是非公平锁）；支持中断响应（lockInterruptibly()）；支持超时获取锁（tryLock()）；可以绑定多个 Condition 条件变量进行精细化线程唤醒。

### **10\. 什么是线程池？四大拒绝策略是什么？Android 中常见线程池有哪些？**

* **答案：** \* **概念**：线程池是一种基于池化思想管理线程的工具，通过复用已创建的线程来降低资源消耗、提高响应速度并提高线程的可管理性。  
  * **四大拒绝策略**：  
    1. AbortPolicy（默认）：直接抛出 RejectedExecutionException 异常。  
    2. CallerRunsPolicy：用调用者所在的线程来执行该任务。  
    3. DiscardOldestPolicy：丢弃阻塞队列中靠最前（最旧）的任务，然后重新尝试执行。  
    4. DiscardPolicy：直接丢弃任务，不给予任何处理或异常。  
  * **常见线程池**：CachedThreadPool（无界可缓存）、FixedThreadPool（定长）、ScheduledThreadPool（定时/周期）、SingleThreadExecutor（单线程）。

### **11\. 简述 ThreadLocal 的原理，以及它为什么可能导致内存泄漏？**

* **答案：** \* **原理**：每个线程内部都维护一个 ThreadLocalMap，其 Key 是 ThreadLocal 对象的弱引用（WeakReference\<ThreadLocal\<?\>\>），Value 是用户真正存储的值。借此实现线程之间的数据隔离。  
  * **内存泄漏原因**：由于 ThreadLocalMap 的 Key 是弱引用，当外部强引用断开后，Key 会在下次 GC 时被回收，此时 Key 变为 null。但 Value 是强引用，只要当前线程不结束（例如在线程池中被长期复用），这条从 CurrentThread \-\> ThreadLocalMap \-\> Entry \-\> Value 的强引用链就会一直存在，导致 Value 内存无法被释放。**解决方法**：使用完后务必手动调用 remove()。

### **12\. Java 中的垃圾回收算法有哪些？简述 G1 收集器。**

* **答案：** \* **基础算法**：标记-清除算法（Mark-Sweep）、标记-复制算法（Copying）、标记-整理算法（Mark-Compact）、分代收集算法。  
  * **G1（Garbage-First）收集器**：面向服务端应用的垃圾收集器。它将整个 Java 堆划分成多个大小相等的独立区域（Region），保留了新生代和老年代的概念，但它们在空间上不再连续。G1 的核心特点是**可预测的停顿时间模型**。它会跟踪各个 Region 里面的垃圾堆积价值，在后台维护一个优先列表，每次根据允许的收集时间，优先回收价值最大的 Region（这也是其名字的由来）。

### **13\. 什么是 CAS（Compare And Swap）？它有什么缺点？如何解决？**

* **答案：** \* **概念**：CAS 是一种无锁的原子操作（乐观锁机制）。它包含三个操作数：内存位置（V）、预期原值（A）和新值（B）。如果内存位置的值与预期原值相匹配，那么处理器会自动将该位置值更新为新值，否则不作任何操作。  
  * **缺点**：  
    1. **ABA 问题**：变量被线程 1 改为 B，又改回 A，线程 2 检查时认为它没变。**解决**：使用版本号或时间戳，如 AtomicStampedReference。  
    2. **循环时间长，开销大**：如果自旋长期不成功，会给 CPU 带来巨大执行开销。  
    3. **只能保证一个共享变量的原子操作**。

### **14\. 泛型中的 \<? extends T\> 和 \<? super T\> 有什么区别？**

* **答案：** \* 符合 **PECS（Producer Extends, Consumer Super）** 原则：  
  * \<? extends T\>（上界通配符）：适用于**生产者（Producer）**，即你只需要从集合中**读取**数据。允许读取出 T 或其父类类型，但禁止向集合中写入任何数据（除了 null）。  
  * \<? super T\>（下界通配符）：适用于**消费者（Consumer）**，即你主要是想向集合中**写入**数据。允许写入 T 及其子类类型的对象，但读取出来的对象只能用 Object 接收。

### **15\. Java 异常体系的结构是怎样的？Error 与 Exception 有何区别？**

* **答案：** \* 基类是 Throwable，其下分为两大子类：Error 和 Exception。  
  * Error（错误）：表示 JVM 无法解决的严重问题，如系统崩溃、虚拟机内存错误（OutOfMemoryError）、栈溢出（StackOverflowError）。应用不应该尝试用 try-catch 捕获它们。  
  * Exception（异常）：程序本身可以处理的异常。分为**编译时异常（Checked Exception，如 IOException）**（必须显式捕获或抛出）和**运行时异常（RuntimeException / Unchecked Exception，如 NullPointerException）**（通常由逻辑漏洞引起，无需强制捕获）。

### **16\. 什么是内部类？匿名内部类为什么能访问局部的 final 变量？**

* **答案：** \* 内部类是定义在另一个类内部的类。分为成员内部类、静态内部类、局部内部类和匿名内部类。  
  * **原因**：因为生命周期不一致（闭包问题）。局部变量存在于栈中，当方法执行完毕后该变量就被销毁了；而匿名内部类对象可能存在于堆中（如作为异步回调）。为了解决这个问题，Java 编译器通过闭包拷贝（Variable Copying）技术，在匿名内部类的构造函数中把这个局部变量复制了一份作为内部类的成员变量。为了保证两边数据的一致性，要求该局部变量必须是 final（Java 8 之后可以是隐式 final）。

### **17\. 解释 Java 中 HashMap 的底层原理（JDK 1.8 之后的变化）。**

* **答案：** \* JDK 1.8 之前采用**数组 \+ 链表**。JDK 1.8 之后引入了红黑树，底层变为**数组 \+ 链表 \+ 红黑树**。  
  * **机制**：当插入键值对时，先计算 Key 的 hash 值并取模定位到数组索引（桶位置）。如果发生冲突，以链表形式挂在后面。当链表长度大于等于 8，且数组总长度大于等于 64 时，链表会转化为**红黑树**，以将查找时间复杂度从 $O(n)$ 降到 $O(\\log n)$；当红黑树节点小于等于 6 时，又会退化回链表。

### **18\. HashMap 是线程安全的吗？多线程下会有什么问题？替代方案是什么？**

* **答案：** \* HashMap 不是线程安全的。  
  * **多线程问题**：在 JDK 1.7 中，多线程并发扩容时由于采用头插法，可能会形成**环形链表**，导致死锁（CPU 100%）；在 JDK 1.8 中改用尾插法虽然避免了死锁，但在多线程下并发写入仍会导致**数据覆盖/丢失**。  
  * **替代方案**：使用 ConcurrentHashMap（并发安全且高效）。不建议使用 Hashtable（全方法加 synchronized 效率极低）。

### **19\. 简述 ConcurrentHashMap 的实现原理（JDK 1.7 与 1.8 的区别）。**

* **答案：** \* **JDK 1.7**：采用分段锁（Segment）技术。将整个大数组拆分为多个 Segment（继承自 ReentrantLock），每把锁只锁住其中一段数据，多线程访问不同段的数据时不存在锁竞争。  
  * **JDK 1.8**：摒弃了 Segment，采用**Node 数组 \+ 链表 \+ 红黑树**的结构。利用 **CAS \+ synchronized** 来保证并发安全。锁的粒度进一步细化，直接锁定链表或红黑树的头节点（桶节点），只要不发生哈希冲突，多线程并发写入完全不会冲突，效率大幅提升。

### **20\. 什么是反射机制？它在 Android 中有哪些应用和代价？**

* **答案：** \* **概念**：在运行状态中，对于任意一个类，都能够知道这个类的所有属性和方法；对于任意一个对象，都能够调用它的任意一个方法和属性。这种动态获取信息及动态调用对象方法的功能称为反射。  
  * **Android 应用**：跨进程通信 Hook、热修复（如动态注入 dex 路径到 dexElements 数组）、各种动态代理框架（如 Retrofit）。  
  * **代价**：性能开销大（涉及类型安全检查、无法享受 JVM 编译优化）；容易破坏封装性，导致维护困难；由于混淆的存在，反射代码容易在发布包中失效（需要配置 keep 规则）。

### **21\. Java 类的生命周期包含哪些阶段？**

* **答案：** \* 包含 7 个阶段：**加载（Loading）** \-\> **验证（Verification）** \-\> **准备（Preparation）** \-\> **解析（Resolution）** \-\> **初始化（Initialization）** \-\> **使用（Using）** \-\> **卸载（Unloading）**。其中验证、准备、解析合并称为连接（Linking）阶段。

### **22\. 什么是死锁？产生死锁的四个必要条件是什么？如何排查？**

* **答案：** \* **概念**：指两个或两个以上的线程在执行过程中，因争夺资源而造成的一种互相等待的现象。  
  * **四个必要条件**：  
    1. **互斥条件**：资源同一时刻只能被一个线程占用。  
    2. **请求与保持条件**：线程因请求新资源而阻塞时，对已获得的资源保持不放。  
    3. **不剥夺条件**：线程已获得的资源在未使用完之前，不能被其他线程强行剥夺。  
    4. **循环等待条件**：存在一个线程等待环路。  
  * **排查**：通过 JVM 工具如 jstack 查看线程堆栈；在 Android 中可通过监测 ANR 日志（traces.txt）或使用底层 Hook 检测锁持有链。

### **23\. Object 类有哪些常用的方法？**

* **答案：** \* equals(Object obj)、hashCode()、toString()、getClass()、clone()、finalize()、wait()、wait(long timeout)、notify()、notifyAll()。

### **24\. 简述 Java 中的浅拷贝与深拷贝。**

* **答案：** \* **浅拷贝（Shallow Copy）**：重新创建了一个对象，但如果原对象的属性是引用类型，拷贝的仅仅是引用地址，新老对象仍共享这部分引用类型的数据（修改会互相影响）。  
  * **深拷贝（Deep Copy）**：不仅重新创建了对象，连对象内部的所有引用类型属性也全部递归地重新创建了一份，新老对象完全独立，互不影响。可通过重写 clone() 逐层拷贝或利用对象序列化/反序列化实现。

### **25\. 为什么说 Java 中的 String 是不可变的？这样做有什么好处？**

* **答案：** \* 因为 String 类被 final 修饰，且其底层存储字符的数组也是 private final 的，没有暴露任何修改该数组的方法。  
  * **好处**：  
    1. **字符串常量池的需要**：可以节省内存，多个相同字面量的引用指向同一个地址。  
    2. **线程安全**：不可变对象天生线程安全，多线程并发访问无需加锁。  
    3. **安全性和哈希缓存**：String 常用作 HashMap 的键，其不可变性确保了 hashCode 永远不变，只需计算一次并缓存，提高效率。同时作为网络传输参数时也更安全。

### **26\. 什么是 AQS（AbstractQueuedSynchronizer）？**

* **答案：** \* AQS 是抽象队列同步器，是构建锁或者其他同步组件（如 ReentrantLock、CountDownLatch、Semaphore）的基础框架。它使用一个 volatile int state 变量表示同步状态，通过 CAS 自旋操作来修改状态，当线程获取锁失败时，会将当前线程包装成一个 Node 节点加入到其内部维护的 FIFO 双向同步队列中，并使其阻塞，等待前驱节点释放锁时将其唤醒。

### **27\. 简述 CountDownLatch 和 CyclicBarrier 的区别。**

* **答案：** \* CountDownLatch 是一组线程等待另一组/一个线程执行完特定操作后再继续执行，其计数器只能使用一次，是一次性的。  
  * CyclicBarrier 是一组线程互相等待，直到所有线程都到达某个屏障点（栅栏）后再同时继续执行，它的计数器可以通过 reset() 重置并循环使用。

### **28\. 什么是 Java 的内存模型（JMM）？**

* **答案：** \* JMM（Java Memory Model）是一组规范，用来屏蔽掉各种硬件和操作系统的内存访问差异，以实现让 Java 程序在各种平台下都能达到一致的内存访问效果。JMM 规定所有的变量都存储在**主内存**中，每条线程还有自己的**工作内存**（保存了主内存变量的副本）。线程对变量的所有操作都必须在工作内存中进行，不能直接读写主内存，线程间变量值的传递均需要通过主内存来完成。

### **29\. 什么是 Java 对象的“半初始化”？**

* **答案：** \* 指在执行 new 指令创建对象时，JVM 首先在堆中为对象分配内存，并将实例变量初始化为各自类型的默认零值（如 int 为 0，引用类型为 null）。此时，对象已经分配了地址，但还没有执行构造方法（\<init\> 指令）来赋予代码中指定的初始值，这个中间状态就称为半初始化状态。

### **30\. 为什么 wait() 和 notify() 方法定义在 Object 类中，而 sleep() 定义在 Thread 类中？**

* **答案：** \* 因为 wait() 和 notify() 是配合内置锁（Monitor）来实现线程间通信的。在 Java 中，任意一个对象都可以作为锁对象，锁是绑定在对象头上的，所以任何对象都应该具备改变锁状态并通知其他等待该锁线程的能力。而 sleep() 是让当前线程让出 CPU 执行权并暂停执行一段时间，这是线程自身的属性和行为，与具体的对象锁无关，因此属于 Thread 类。

## **第二部分：Android 核心组件与系统机制（第 31-60 题）**

### **31\. 请详细描述 Android 系统的启动流程。**

* **答案：** 1\. **Power On**：通电，引导芯片代码（BootLoader）开始执行。  
  2\. **Kernel 启动**：加载 Linux 内核，初始化硬件驱动，并启动 init 进程（Linux 的 1 号进程）。  
  3\. **init 进程**：解析 init.rc 配置文件，孵化出重要守护进程（如 installd、servicemanager），并启动 Zygote 进程。  
  4\. **Zygote 进程**：创建 Java 虚拟机，预加载 Android 核心类和资源，随后通过 fork 自身孵化出 SystemServer 进程。  
  5\. **SystemServer 进程**：启动 Android 的所有核心系统服务（如 ActivityManagerService (AMS)、WindowManagerService (WMS)、PackageManagerService (PMS) 等），并将它们注册到 ServiceManager 中。  
  6\. **Launcher 启动**：AMS 完成初始化后，发送 Intent 启动桌面程序（Launcher），展示手机主界面。

### **32\. 什么是 Zygote 进程？为什么 Android 要用它来孵化进程，而不是直接 fork init 进程？**

* **答案：** \* **概念**：Zygote 是 Android 系统中所有 Java 进程的鼻祖。  
  * **原因**：Android 应用运行需要 Java 虚拟机（ART/Dalvik）环境及大量的系统核心类与基础资源。Zygote 在启动时会把这些虚拟机环境、核心类库（Framework 层）和常用资源全部加载到内存中。当需要启动新 App 时，Zygote 通过 Linux 的 fork() 机制（配合 **写时复制 Copy-on-Write** 技术）直接复制自身。这样新进程瞬间就拥有了完整的虚拟机和预加载资源，**极大地缩短了 App 启动时间，并实现了多个进程共享同一份核心类库内存，显著节省了系统整体内存开销**。

### **33\. 谈谈 Dalvik 虚拟机与 ART 虚拟机的区别（JIT 与 AOT）。**

* **答案：** \* **Dalvik VM**：采用 **JIT（Just-In-Time）即时编译**。App 运行时，解释器将字节码实时翻译成机器码，每次运行都需要翻译，导致运行效率低、耗电。  
  * **ART（Android Runtime）**：  
    * **早期（Android 5.0 \- 6.0）**：采用 **AOT（Ahead-Of-Time）预编译**。应用在安装时就将字节码全量编译成机器码，App 运行速度大幅提升（空间换时间，导致安装慢且占用空间大）。  
    * **现代优化（Android 7.0+）**：采用 **混合编译模式（JIT \+ AOT \+ 配置文件引导）**。安装时不进行预编译以加快安装速度；在运行时通过 JIT 编译并记录热点代码；当设备处于空闲且充电状态时，系统会根据记录的热点代码在后台进行 AOT 编译。完美平衡了安装速度、存储空间和运行性能。

### **34\. 简述 Activity 的生命周期，并说明 A Activity 启动 B Activity 时，两者的生命周期执行顺序。**

* **答案：** \* **典型生命周期**：onCreate() \-\> onStart() \-\> onResume() \-\> onPause() \-\> onStop() \-\> onDestroy()。在异常情况下可能触发 onRestart()、onSaveInstanceState() 和 onRestoreInstanceState()。  
  * **启动顺序**：A 启动 B 时的生命周期严格遵循以下顺序：  
    1. A.onPause()  
    2. B.onCreate()  
    3. B.onStart()  
    4. B.onResume()  
    5. A.onStop()（前提是 B 完全遮挡了 A。如果 B 是透明或对话框主题，A 绝不会触发 onStop()）。  
  * **面试官追问提示**：因此，绝不能在 A 的 onPause() 中做耗时操作，否则会直接拖慢 B 的启动速度。

### **35\. Activity 的四种启动模式（LaunchMode）分别是什么？有什么应用场景？**

* **答案：** 1\. **standard（标准模式，默认）**：每次启动都会创建一个全新的实例压入当前任务栈。  
  2\. **singleTop（栈顶复用模式）**：如果要启动的 Activity 已经位于任务栈的栈顶，则不创建新实例，而是直接复用栈顶实例，并回调其 onNewIntent() 方法。**场景**：推送通知点击跳转的页面、浏览器书签页。  
  3\. **singleTask（栈内复用模式，单例）**：只要 Activity 在栈中存在，启动它时就会把它上方的所有 Activity 清除出栈（Clear Top），使其位于栈顶，并回调 onNewIntent()。**场景**：App 主页（MainActivity）。  
  4\. **singleInstance（单实例模式）**：极其严格。该 Activity 会独自占用一个全新的任务栈，且该任务栈内不允许有其他 Activity 存在。**场景**：来电通话界面、闹钟提醒界面。

### **36\. 简述 Service 的启动方式：startService 与 bindService 的生命周期和区别。**

* **答案：** \* **startService**：生命周期为 onCreate() \-\> onStartCommand() \-\> onDestroy()。Service 与调用者没有强绑定关系，调用者销毁了，Service 依然在后台运行，必须手动调用 stopService 或 stopSelf 来停止。  
  * **bindService**：生命周期为 onCreate() \-\> onBind() \-\> onUnbind() \-\> onDestroy()。Service 与调用者（如 Activity）强绑定，通信依赖返回的 IBinder 对象。调用者一旦销毁，Service 会自动解绑并销毁。  
  * **混合使用**：若先 start 再 bind，需要同时调用 unbindService 和 stopService 才能彻底销毁该 Service。

### **37\. 什么是 IntentService？它与普通 Service 有什么区别？（注：高版本推荐用 WorkManager / JobIntentService 替代）**

* **答案：** \* 普通 Service 默认运行在**主线程**中，不能直接做耗时操作。  
  * IntentService 是 Service 的子类，其内部封装了 HandlerThread（一个串行工作的工作线程）和 Handler。当通过 startService 提交任务时，它会自动将任务推入工作线程的队列中，在 onHandleIntent() 中**异步串行处理**。所有任务处理完毕后，IntentService 会**自动调用 stopSelf() 销毁自己**，无需手动停止。

### **38\. 简述 BroadcastReceiver（广播）的注册方式及区别。**

* **答案：** \* **静态注册**：在 AndroidManifest.xml 中配置。特点是**常驻型**，即便应用未启动，一旦有匹配的广播发出，系统也会自动唤醒该应用并处理。由于耗电和安全问题，Android 高版本对其添加了极大限制（禁止了绝大多数隐式广播）。  
  * **动态注册**：在代码中调用 registerReceiver() 绑定。特点是**非常驻型**，生命周期与宿主（如 Activity/Context）绑定，宿主销毁前必须显式调用 unregisterReceiver() 解绑，否则会导致内存泄漏。

### **39\. ContentProvider（内容提供者）的作用和原理是什么？**

* **答案：** \* **作用**：主要用于**跨应用（进程间）安全地共享数据**（如读取系统通讯录、多媒体库）。  
  * **原理**：其底层核心是基于 **Binder IPC 机制**。ContentProvider 将底层的数据源（SQLite 数据库、文件、网络等）抽象成类似数据库表结构的统一 URI 接口。外部应用通过 ContentResolver，传入特定的 URI、projection 等参数，通过 Binder 跨进程调用 ContentProvider 的 query、insert 等方法完成数据交互。

### **40\. 详细阐述 Android 中的事件分发机制。**

* **答案：** \* 事件分发的核心对象是 MotionEvent，分发路径遵循“树状结构”，主要涉及三个核心方法：  
  1\. dispatchTouchEvent(MotionEvent ev)：用于事件的分发。只要事件传递到当前 View，该方法必定触发。  
  2\. onInterceptTouchEvent(MotionEvent ev)：ViewGroup 独有方法，用于判断**是否拦截事件**。返回 true 则拦截，交给自身的 onTouchEvent 处理；返回 false 则向下传递给子 View。  
  3\. onTouchEvent(MotionEvent ev)：用于处理事件。返回 true 表示消费该事件，事件流终止；返回 false 表示不消费，事件将**逆向向上回溯**传给父布局的 onTouchEvent 处理。  
  * **分发总链条**：Activity \-\> PhoneWindow \-\> DecorView \-\> ViewGroup \-\> View。若到底层 View 均不消费，则逆向回溯：View \-\> ViewGroup \-\> DecorView \-\> Activity。

### **41\. 什么是 View 的绘制流程？主要经历哪三大步骤？**

* **答案：** \* View 的绘制流程是从 ViewRootImpl 的 performTraversals() 开始的，从上往下遍历整个 View 树，三大核心步骤为：  
  1\. **Measure（测量）**：确定 View 的测量宽高（onMeasure()）。根据父容器的 MeasureSpec 约束和自身的 LayoutParams 共同决定。  
  2\. **Layout（布局）**：确定 View 在父容器中的具体放置坐标（onLayout()）。ViewGroup 负责计算并分配子 View 的 l、t、r、b 位置。  
  3\. **Draw（绘制）**：将 View 绘制到屏幕上（onDraw()）。包含绘制背景、绘制自身内容、绘制子 View（dispatchDraw()）、绘制装饰（如滚动条）等。

### **42\. invalidate() 和 requestLayout() 的区别是什么？**

* **答案：** \* invalidate()：只会触发 **Draw（重新绘制）**。当 View 的外观、颜色等发生改变但大小和位置没变时调用。它会向上传递脏区域标记，最终导致 onDraw() 被重新调用，不会触发布局测量。  
  * requestLayout()：会触发完整的 **Measure（测量）和 Layout（布局）** 流程，但不一定会触发 Draw（除非大小改变导致需要重绘）。当 View 的尺寸、边距发生改变，需要重新计算空间大小时调用。

### **43\. 谈谈 Android 中的 Binder 机制，为什么它如此高效？**

* **答案：** \* Binder 是 Android 系统的 IPC（进程间通信）核心架构。它基于 Linux 的内存映射（mmap()）技术。  
  * **传统 Linux IPC**（如管道/消息队列）：数据需要经历两次拷贝（发送方用户空间 \-\> 内核缓存区 \-\> 接收方用户空间）。  
  * **Binder 机制**：只需要**一次拷贝**。Binder 驱动通过 mmap()，在内核空间中开辟一块缓冲区，并将这块内核缓冲区直接映射到接收方进程的用户空间。这样，发送方只需将数据从自己的用户空间拷贝到内核缓冲区，接收方由于内存映射关系，可以立刻“看见”并读取数据。不仅高效，而且 Binder 基于 UID/PID 的身份校验机制，安全性极高。

### **44\. 什么是 ANR（Application Not Responding）？引发 ANR 的根本原因和排查手段是什么？**

* **答案：** \* **概念**：应用无响应。  
  * **根本原因**：主线程（UI线程）被耗时操作阻塞，导致无法及时响应用户操作或系统回调。  
  * **典型触发阈值**：  
    1. 输入事件（点击、触摸）超过 5 秒未响应。  
    2. BroadcastReceiver 的 onReceive() 在前台 10 秒（后台 60 秒）内未执行完毕。  
    3. Service 的生命周期回调在前台 20 秒（后台 200 秒）内未处理完。  
  * **排查手段**：提取并分析系统生成的 ANR 堆栈日志文件（高版本通过 Storage 或 /data/anr/traces.txt）；利用 Android Studio 的 Profiler 观察 CPU 线程死锁情况；线上引入 StrictMode（严苛模式）或 BlockCanary 库进行主线程卡顿监控。

### **45\. 什么是内存泄漏（Memory Leak）？Android 中有哪些常见的内存泄漏场景？**

* **答案：** \* **概念**：长生命周期的对象持有短生命周期对象的强引用，导致短生命周期对象在任务结束后无法被垃圾回收器（GC）释放，白白占用内存。  
  * **常见场景**：  
    1. **静态变量持有 Context/Activity**。  
    2. **非静态内部类/匿名内部类**（如非静态 Handler、Runnable、异步任务）：它们隐式持有外部类（Activity）的引用，若异步任务在 Activity 销毁时未结束，会导致 Activity 泄漏。  
    3. **单例模式**：传入了 Activity 的 Context 导致单例长期持有该 Activity（应使用 Context.getApplicationContext()）。  
    4. **资源未关闭**：如 Cursor、File 流、Bitmap 未调用 recycle()、各种 Listener/广播未解绑。

### **46\. 如何在 Android 中正确使用 Handler 并防止内存泄漏？**

* **答案：** \* **原因**：非静态内部类 Handler 隐式持有外部 Activity 的引用。当发送延时消息时，Message 存放在主线程的 MessageQueue 中，Message.target 指向该 Handler，而 Handler 又持有 Activity。导致 Activity 销毁时强引用链不释放。  
  * **防泄漏标准写法**：  
    1. 将 Handler 声明为**静态内部类（Static Class）**（破坏隐式引用）。  
    2. 在静态 Handler 内部，通过弱引用（WeakReference）来持有和访问外部的 Activity。  
    3. 在 Activity 的 onDestroy() 生命周期方法中，调用 handler.removeCallbacksAndMessages(null) 清空消息队列中的所有等待消息和任务。

### **47\. 简述 Android 性能优化中的“屏幕墙”或“卡顿（Jank）”产生的原因。**

* **答案：** \* Android 系统每隔 16.6ms（基于 60Hz 刷新率）会发出一个 **Choreographer VSYNC（垂直同步）信号**。收到信号后，系统开始进行 CPU 计算（Measure/Layout/Draw）和 GPU 渲染，并将数据写入帧缓冲区。如果在 16.6ms 内，主线程因为复杂的布局计算、主线程做耗时 I/O 操作或过度绘制（Overdraw）没有及时完成这一帧的准备工作，当 VSYNC 信号到来时，屏幕只能继续显示上一帧的内容，这就造成了一次丢帧，用户主观感受就是卡顿（Jank）。

### **48\. 什么是“过度绘制（Overdraw）”？如何优化？**

* **答案：** \* **概念**：屏幕上的某个像素在同一帧的时间内被重复绘制了多次。这浪费了大量的 GPU 渲染带宽。  
  * **优化手段**：  
    1. 在开发者选项中开启“调试 GPU 过度绘制”，观察颜色分布（移除红、紫色区域）。  
    2. **移除冗余的背景**：去掉布局中不必要的 android:background（如 Activity 已有全局背景，子布局就不用再设相同背景）。  
    3. **扁平化布局结构**：使用 ConstraintLayout（约束布局）减少嵌套层级，或配合 \<merge\>、\<viewstub\> 标签按需加载。  
    4. 自定义 View 时，使用 canvas.clipRect() 限制只绘制可见区域。

### **49\. 简述组件化开发（Componentization）的核心思想及架构设计。**

* **答案：** \* **核心思想**：将一个庞大的 App 按照业务逻辑或功能模块拆分为多个彼此独立的“组件”（Module）。每个组件在开发阶段可以独立作为 Application 运行调试，在打包发布阶段可以作为 Library 组装沉淀。  
  * **架构设计**：  
    1. **壳工程（App）**：负责集成各业务组件，配置主入口。  
    2. **业务组件层（Feature Modules）**：如登录组件、商城组件、个人中心。彼此之间互不依赖，实现彻底解耦。  
    3. **功能组件/通用库层（Core/Common Modules）**：网络库、图片加载库、基类库，供上层业务组件调用。  
    4. **跨组件通信**：由于业务组件间无依赖，必须通过路由框架（如 ARouter）进行页面跳转和接口下发。

### **50\. 谈谈你对 MVC、MVP 和 MVVM 架构模式的理解与区别。**

* **答案：** \* **MVC**：Model（数据）- View（布局）- Controller（Activity/Fragment）。缺点是 Activity 既做控制又做 View 渲染，导致代码臃肿，层级职责划分不清。  
  * **MVP**：Model \- View \- Presenter。Presenter 充当纽带，**隔离了 View 和 Model**。View 和 Presenter 之间通过定义好的**接口**双向通信。缺点是会产生大量的接口类，且 Presenter 容易随业务扩展而膨胀，同时容易引发内存泄漏（Presenter 持有 View 引用）。  
  * **MVVM**：Model \- View \- ViewModel。核心是**数据驱动（Data Binding / Jetpack LiveData/ViewModel）**。ViewModel 内部不持有任何 View 的引用。View 订阅 ViewModel 的状态（数据），当数据发生变化时，UI 会自动刷新。实现了真正的解耦和关注点分离。

### **51\. 什么是 LocalBroadcastManager？它与普通 BroadcastReceiver 有何区别？（注：高版本推荐用 Flow/SharedFlow 替代）**

* **答案：** \* 普通广播是全局跨进程的，发出的广播所有应用都可以拦截，存在严重的安全隐患，且底层走的是 Binder IPC 通信，有性能开销。  
  * LocalBroadcastManager 是**应用内本地广播**。它底层完全没有走 Binder，而是直接在应用内部基于 **Handler 机制和观察者模式** 来实现的。因此它极度安全（外部无法监听），且运行速度非常快。

### **52\. 谈谈你对 Android 中 Serializable 和 Parcelable 序列化接口的理解和选择。**

* **答案：** \* **Serializable**：Java 自带的序列化接口。使用非常简单（只需实现空接口），但它是一个**重量级**的操作。底层基于大量的**I/O 读写和反射机制**，在运行过程中会产生大量的临时对象，效率低。**场景**：适合网络传输或持久化存储到磁盘中。  
  * **Parcelable**：Android 特有的序列化接口。实现较为繁琐（需要手动写偏序代码，虽有插件辅助）。它的原理是在内存中进行数据打包，直接利用底层高效的内存结构读写，**不走反射，效率高出 Serializable 十倍以上**。**场景**：适合在 Android 进程间（Binder）或 Intent 传参等内存通信场景中首选。

### **53\. 为什么 Activity 在旋转屏幕后会重启？如何防止它重启？**

* **答案：** \* **原因**：旋转屏幕属于**系统配置发生改变（Configuration Changes）**。为了确保应用能够正确加载最匹配当前屏幕方向的资源（如 layout-land 横屏布局），系统默认会销毁当前的 Activity 并重新创建一个新的实例。  
  * **防重启方法**：在 AndroidManifest.xml 中为对应的 \<activity\> 节点配置 android:configChanges="orientation|screenSize|keyboardHidden" 属性。配置后，屏幕旋转时系统不再销毁重启 Activity，而是直接回调该 Activity 的 onConfigurationChanged() 方法。

### **54\. 简述 ViewStub 的作用和工作原理。**

* **答案：** \* **作用**：ViewStub 是一个不可见的、零大小的占位视图，用于实现布局的**按需延迟加载（Lazy Load）**。  
  * **原理**：当布局加载时，ViewStub 占用极小的系统资源。只有当主动调用其 inflate() 方法或将其可见性设置为 View.VISIBLE 时，ViewStub 才会开始加载其指向的真实目标布局，并将目标布局替换到 ViewStub 当前在父容器中的位置。非常适合处理如“网络错误提示页”、“加载失败页”等非常规显示界面。

### **55\. 什么是 APK 的瘦身（优化大小）？你有哪些常用手段？**

* **答案：** \* APK 主要由 dex、res（资源）、assets、lib（so库）和 resources.arsc 组成。  
  * **常用手段**：  
    1. **开启代码混淆与资源缩减**：配置 minifyEnabled true 和 shrinkResources true 自动剔除无用代码和未引用资源。  
    2. **图片优化**：将传统 PNG/JPG 转换为 **WebP** 格式；大图尽量使用 VectorDrawable（矢量图）。  
    3. **SO 库裁剪**：通过 ndk.abiFilters 只配置当前主流的架构（如 armeabi-v7a 或 arm64-v8a），不打包全架构。  
    4. **资源在线化**：不常用的音视频、字体或大图资产放入服务器，使用时动态下载。

### **56\. 简述 LruCache 的底层原理。**

* **答案：** \* LruCache（Least Recently Used，最近最少使用算法）常用于内存缓存（如 Bitmap 缓存）。  
  * **底层原理**：它内部封装了一个 **LinkedHashMap**，并将其构造参数的 accessOrder 设置为 true（这意味着每次访问/get 某个元素时，该元素都会被自动移动到链表的末尾）。当向 LruCache 中 put 新数据时，它会计算当前缓存总容量是否超过了预设的最大限制值。如果超限，就会在循环中不断地把链表头部（即最久未被访问过的、最老的元素）移除掉，直到容量降到安全线以下。

### **57\. 什么是 Fragment 的“状态丢失（Can not perform this action after onSaveInstanceState）”异常？如何解决？**

* **答案：** \* **原因**：当系统内存不足或 Activity 压入后台时，系统会触发 onSaveInstanceState() 保存当前的 UI 状态。如果在这个生命周期执行完之后，应用在异步回调中尝试去 commit() 提交一个 FragmentTransaction，系统为了防止这部分最新的提交无法被保存而导致状态丢失，就会抛出该崩溃异常。  
  * **解决方法**：  
    1. 尽量不在异步回调中进行 Fragment 提交。  
    2. 确保在 Activity 处于活跃状态（如 onResume 之后）再提交。  
    3. 如果对状态丢失不敏感（如仅仅是个提示弹窗），可以使用 commitAllowingStateLoss() 替代标准的 commit()。

### **58\. 什么是 SurfaceView？它与普通 View 有什么区别？**

* **答案：** \* 普通 View 都是在系统的主线程中共享同一个 window 绘图表面进行渲染的，如果刷新频率过高或计算复杂，极易导致主线程卡顿。  
  * SurfaceView 内部拥有一个**独立的、受底层系统管理的绘图表面（Surface）**。它允许开发者在**非主线程（子线程）中进行画布 Canvas 锁定制图并进行独立的高频渲染**。非常适合用于游戏开发、视频播放器、相机预览等需要高帧率和高吞吐量图形处理的场景。

### **59\. 简述 Android 7.0、8.0、9.0、10.0+ 等高版本系统较为瞩目的适配要点。**

* **答案：** \* **7.0 (API 24\)**：引入 FileProvider，严禁在应用外部暴露 file:// 类型的 URI，必须转为 content://。  
  * **8.0 (API 26\)**：通知渠道（Notification Channels）适配；限制后台服务（不能直接 startService，需用 startForegroundService）。  
  * **9.0 (API 28\)**：全面限制非 HTTP（明文网络流量请求），强制推行 HTTPS（需配置安全配置文件）；限制对非公开 API（隐藏类和方法）的反射调用。  
  * **10.0+ (API 29+)**：推行**分区存储（Scoped Storage）**，App 无法直接利用绝对路径任意读写外部 SD 卡公共目录，必须走沙盒路径或 SAF（存储访问框架）。

### **60\. 什么是 Jetpack Lifecycle 组件？它解决了什么问题？**

* **答案：** \* Lifecycle 是 Android Jetpack 的核心基础组件。  
  * **解决问题**：在传统开发中，如果某个业务类（如视频播放器、地理定位器）需要根据 Activity 的生命周期进行初始化和释放（如 onStart 开始定位，onStop 停止），我们必须在 Activity 对应的生命周期函数里手动显式调用它们。这导致 Activity 异常臃肿，业务高耦合。  
  * **原理**：Lifecycle 引入了**生命周期所有者（LifecycleOwner）和生命周期观察者（LifecycleObserver）概念。业务类只需实现 DefaultLifecycleObserver 并通过注解或方法绑定生命周期事件，接着在 Activity 中注册该观察者即可。业务类就能够自动感应、自主处理**宿主生命周期的变化，实现了高内聚低耦合的代码组织。
