### **第一部分：Java 基础、并发与 JVM（第 1-40 题）**

#### **1\. 简述 Java 中继承、接口和装饰器模式的区别。**

* **答案**：  
  * **继承（Inheritance）**：建立 “Is-A”（是一个）的关系。子类自动拥有父类的属性和方法，核心目的是代码复用，属于强耦合。  
  * **接口（Interface）**：建立 “Like-A”（具备某种能力）的关系。它是一种完全抽象的契约或规范，只定义方法不实现逻辑，核心目的是解耦。  
  * **装饰器模式（Decorator）**：建立 “Has-A”（组合）的关系。通过将对象嵌入到装饰器对象中，在不改变原有类结构和继承体系的情况下，动态地为对象扩展新功能（增强/换肤）。

#### **2\. JVM 的运行时数据区包含哪些部分？哪些是线程私有的？**

* **答案**：包含程序计数器（Program Counter Register）、Java 虚拟机栈（JVM Stack）、本地方法栈（Native Method Stack）、Java 堆（Heap）、方法区（Method Area）以及运行时常量池。其中，**程序计数器、虚拟机栈、本地方法栈**是线程私有的；**Java 堆、方法区**是线程共享的。

#### **3\. 什么是强引用、软引用、弱引用和虚引用？**

* **答案**：  
  * **强引用（StrongReference）**：普通的正常对象引用，只要强引用存在，垃圾回收器永远不会回收该对象。  
  * **软引用（SoftReference）**：有用但非必须的对象，在系统将要发生内存溢出（OOM）之前，会被回收。常用于缓存。  
  * **弱引用（WeakReference）**：只要进行垃圾回收（GC），无论内存是否足够，都会被回收。Android 中常用于防止 Handler 等引起的内存泄漏。  
  * **虚引用（PhantomReference）**：不对对象生命周期产生任何影响，必须和引用队列（ReferenceQueue）联合使用，主要用于跟踪对象被垃圾回收的活动。

#### **4\. 简述 Java 虚拟机的类加载机制和双亲委派模型。**

* **答案**：类加载机制是指虚拟机把描述类的数据从 Class 文件加载到内存，并对数据进行校验、解析和初始化。  
  **双亲委派模型**是指当一个类加载器收到类加载请求时，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成，只有当父加载器反馈自己无法完成这个加载请求时，子加载器才会尝试自己去加载。这保证了 Java 核心 API 的安全性和唯一性。

#### **5\. 什么是内存泄漏？Android 中常见的内存泄漏有哪些？**

* **答案**：内存泄漏是指程序中已动态分配的堆内存由于某种原因未释放或无法释放，造成系统内存的浪费。Android 中常见原因包括：静态变量持有 Activity 引用、非静态内部类/匿名内部类（如 Handler、Thread、AsyncTask）持有外部 Activity 引用且生命周期长于 Activity、未取消注册的广播接收器或 EventBus、未关闭的资源（Cursor、File、Stream、Bitmap）。

#### **6\. 深入理解 volatile 关键字的作用及原理。**

* **答案**：  
  * **可见性**：当一个线程修改了 volatile 变量的值，新值对于其他线程是立即成可见的。底层是通过在写操作后加入 Store 屏障，强行将本地内存刷新回主内存。  
  * **禁止指令重排序**：编译器和处理器在编译和执行字节码时为了提高性能可能会改变指令顺序，volatile 通过在生成字节码时插入内存屏障（Memory Barrier）来禁止这种优化。  
  * *注意*：volatile 不能保证原子性。

#### **7\. 为什么单例模式的双重检查锁定（DCL）中，instance 变量必须加 volatile？**

* **答案**：防止编译器和处理器进行指令重排序。因为 instance \= new Singleton() 并非原子操作，在底层包含三个步骤：1. 分配内存空间；2. 初始化对象；3. 将引用指向分配的内存。如果不加 volatile，步骤 2 和 3 可能会颠倒。这会导致另一个线程在第一次判空时，拿到一个“已分配内存但未初始化完成”的半成品对象，从而引发空指针异常。

#### **8\. 简述 HashMap 的底层实现原理及扩容机制（Java 8）。**

* **答案**：Java 8 中 HashMap 底层是基于**数组+链表+红黑树**实现的。当链表长度大于等于 8 且数组长度大于等于 64 时，链表会转换为红黑树以提高查询效率。  
  **扩容机制**：默认初始容量为 16，加载因子为 0.75。当 HashMap 中的元素个数超过 容量 \* 加载因子 时就会触发扩容，每次扩容为原来的 2 倍。

#### **9\. HashMap 与 ConcurrentHashMap 的区别？**

* **答案**：HashMap 是线程不安全的，在并发环境下可能导致死循环（Java 7 链表成环）或数据丢失。ConcurrentHashMap 是线程安全的，Java 7 采用**分段锁（Segment）技术，Java 8 放弃了分段锁，改为使用CAS \+ synchronized**来保证并发安全，锁的粒度细化到了每个数组桶（Node）的头节点，性能大幅提升。

#### **10\. synchronized 与 ReentrantLock 的区别？**

* **答案**：  
  * **实现层面**：synchronized 是 Java 关键字，依赖于 JVM 层面（底层是 monitor 机制）；ReentrantLock 是 JDK 提供的 API（基于 AQS 框架）。  
  * **锁的释放**：synchronized 执行完代码块或发生异常时自动释放锁；ReentrantLock 必须手动在 finally 块中调用 unlock()。  
  * **灵活性**：ReentrantLock 提供了更多高级功能：可以实现公平锁、能响应中断（lockInterruptibly）、能尝试获取锁（tryLock），以及支持绑定多个 Condition 条件。

#### **11\. 简述 AQS（AbstractQueuedSynchronizer）的原理。**

* **答案**：AQS 是一个抽象队列同步器，是构建锁或者其他同步组件（如 ReentrantLock、CountDownLatch、Semaphore）的基础框架。它内部使用了一个 volatile int state 状态变量表示同步状态，并通过一个 FIFO 的双向队列（CLH 队列）来完成资源获取线程的排队工作。利用 CAS 操作来安全地修改 state 状态。

#### **12\. 什么是线程池？为什么要使用线程池？**

* **答案**：线程池是一种线程使用模式，它维护着多个线程，等待着监督管理者分配可并发执行的任务。  
  **原因**：1. 降低资源消耗（通过复用已创建的线程，减少线程创建和销毁造成的消耗）；2. 提高响应速度（任务到达时不需要等待线程创建）；3. 提高线程的可管理性（防止无限制创建线程导致耗尽系统内存）。

#### **13\. 线程池的核心参数有哪些？拒绝策略有哪些？**

* **答案**：  
  * **核心参数**：corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（空闲线程存活时间）、unit（时间单位）、workQueue（任务等待队列）、threadFactory（线程工厂）、handler（拒绝策略）。  
  * **拒绝策略**：AbortPolicy（直接抛出异常，默认）、CallerRunsPolicy（用调用者所在的线程来执行任务）、DiscardOldestPolicy（丢弃队列中最靠前的任务并尝试再次提交）、DiscardPolicy（直接丢弃任务，不予处理）。

#### **14\. 简述 Java 垃圾回收（GC）中如何判断一个对象可以被回收？**

* **答案**：主要有两种方法：  
  * **引用计数法**：给对象添加一个引用计数器，每当有一个地方引用它时计数器加 1，引用失效时减 1。缺点是无法解决对象间循环引用的问题。  
  * **可达性分析算法（主流）**：从一系列被称为 “GC Roots” 的根对象开始向下搜索，搜索所走过的路径称为引用链。当一个对象到 GC Roots 没有任何引用链相连（即不可达）时，则证明此对象是不可用的，可以被回收。

#### **15\. 哪些对象可以作为 GC Roots？**

* **答案**：1. 虚拟机栈（栈帧中的本地变量表）中引用的对象；2. 方法区中类静态属性引用的对象；3. 方法区中常量引用的对象；4. 本地方法栈中 JNI（即 Native 方法）引用的对象。

#### **16\. 简述 JVM 常见的垃圾收集算法。**

* **答案**：  
  * **标记-清除算法（Mark-Sweep）**：分为标记和清除两个阶段。缺点是会产生大量不连续的内存碎片。  
  * **标记-复制算法（Copying）**：将可用内存按容量划分为大小相等的两块，每次只使用其中一块。发生 GC 时将存活对象复制到另一块，清空当前块。缺点是内存利用率低（通常只有 50%）。  
  * **标记-整理算法（Mark-Compact）**：标记后让所有存活对象都向内存空间的一端移动，然后直接清理掉边界以外的内存。适合老年代。  
  * **分代收集算法**：根据对象存活周期的不同将内存划分为新生代和老年代。新生代用复制算法，老年代用标记-清除或标记-整理算法。

#### **17\. 简述 String、StringBuilder 和 StringBuffer 的区别。**

* **答案**：  
  * String：不可变的字符序列，每次对其进行修改都会产生新的对象，性能较低。  
  * StringBuilder：可变的字符序列，线程不安全，单线程下字符串拼接效率最高。  
  * StringBuffer：可变的字符序列，内部方法使用了 synchronized 关键字，线程安全，效率比 StringBuilder 稍低。

#### **18\. Java 中的泛型擦除是什么？**

* **答案**：Java 的泛型是伪泛型。泛型信息只存在于代码编译阶段，在进入 JVM 之前，编译期会将所有的泛型信息擦除掉（Erased），并将其替换为原始类型（如 Object，或者其限定的上界），并在读取时插入强制类型转换。因此，在运行时无法通过反射获取泛型的具体类型参数（如无法区分 List\<String\> 和 List\<Integer\> 的 Class 类型）。

#### **19\. 什么是反射？反射有什么优缺点？**

* **答案**：反射是在运行状态中，对于任意一个类，都能够知道这个类的所有属性和方法；对于任意一个对象，都能够调用它的任意一个方法和属性。  
  * **优点**：运行期类型的动态判断、动态加载，极大地提高了代码的灵活性（如各种框架和依赖注入的实现）。  
  * **缺点**：性能开销大（需要解析字节码）；安全问题（可以破坏封装性访问私有成员）。

#### **20\. 简述 Java 中的异常体系。**

* **答案**：所有异常都继承自 Throwable。它有两个主要子类：  
  * Error：严重问题，JVM 无法处理，如 OutOfMemoryError、StackOverflowError，程序不该试图用 catch 捕获。  
  * Exception：程序本身可以处理的异常。分为**运行时异常（RuntimeException / Unchecked）**（如 NullPointerException，编译时不强制处理）和**编译时异常（Checked）**（如 IOException，编译时强制要求必须 try-catch 或 throws 声明）。

#### **21\. equals() 和 hashCode() 的关系是什么？为什么重写 equals 必须重写 hashCode？**

* **答案**：如果两个对象通过 equals() 比较相等，那么它们的 hashCode() 值也必须相同。  
  如果重写了 equals 而不重写 hashCode，当这些对象被作为 Key 存入 HashMap 或 HashSet 时，会导致两个“逻辑上相等”的对象因为 hashCode 不同而被散列到不同的桶中，导致无法正确去重或获取，违背了集合的契约。

#### **22\. 什么是内部类？非静态内部类为什么会持有外部类的引用？**

* **答案**：内部类是定义在另一个类内部的类。非静态内部类（包括匿名内部类）在编译后，编译器会自动为其生成一个构造函数，并将外部类的引用作为参数传入该构造函数并保存。因此，非静态内部类会隐式持有外部类的引用。静态内部类则不会持有。

#### **23\. 简述 ThreadLocal 的原理以及内存泄漏隐患。**

* **答案**：ThreadLocal 用于实现线程间的隔离，每个线程内部持有一个 ThreadLocalMap 映射表，其 Key 是 ThreadLocal 对象的弱引用（WeakReference\<ThreadLocal\<?\>\>），Value 是真正要存储的值。  
  **内存泄漏原因**：由于 Key 是弱引用，一旦 ThreadLocal 对象没有强引用时会被 GC 回收，此时 Key 变为 null，但 Value 是强引用，只要线程不结束，这条 null \-\> Value 的引用链就会一直存在，导致 Value 内存泄露。解决办法是每次使用完手动调用 remove()。

#### **24\. 什么是死锁？产生死锁的四个必要条件是什么？**

* **答案**：死锁是指两个或两个以上的线程在执行过程中，因争夺资源而造成的一种互相等待的现象。  
  **必要条件**：1. 互斥条件（资源只能被一个线程占用）；2. 请求与保持条件（线程因请求资源而阻塞时，对已获得的资源保持不放）；3. 不剥夺条件（线程已获得的资源在未使用完之前不能被强行剥夺）；4. 循环等待条件（若干线程之间形成一种头尾相接的循环等待资源关系）。

#### **25\. 如何避免死锁？**

* **答案**：破坏死锁的四个必要条件之一：1. 破坏请求与保持条件（一次性申请所有资源）；2. 破坏不剥夺条件（占用部分资源的线程进一步申请其他资源受阻时，主动释放已占有的资源）；3. 破坏循环等待条件（按顺序加锁，使所有线程按照相同的顺序请求资源，或使用 ReentrantLock.tryLock() 设定超时时间）。

#### **26\. 简述 Java 中的轻量级锁、偏向锁、重量级锁（锁升级）。**

* **答案**：Java 6 为了减少获得锁和释放锁带来的性能消耗，引入了偏向锁和轻量级锁。锁升级的状态顺序为：**无锁 \-\> 偏向锁 \-\> 轻量级锁 \-\> 重量级锁**，锁只能升级不能降级。  
  * **偏向锁**：同一线程反复获取锁时，消除同步，只需比对 Mark Word 中的线程 ID。  
  * **轻量级锁**：当出现多个线程竞争锁但没有竞争冲突时，通过 CAS 自旋来获取锁，避免线程阻塞。  
  * **重量级锁**：当自旋超过一定次数或多线程激烈竞争时，锁升级为重量级锁，底层通过操作系统的互斥量（Mutex）实现，未获取锁的线程会进入阻塞状态。

#### **27\. 面向对象的设计原则有哪些（SOLID）？**

* **答案**：  
  * **S**（单一职责原则）：一个类应该只有一个引起它变化的原因。  
  * **O**（开闭原则）：对扩展开放，对修改关闭。  
  * **L**（里氏替换原则）：所有引用基类的地方必须能透明地使用其子类的对象。  
  * **I**（接口隔离原则）：客户端不应该依赖它不需要的接口。  
  * **D**（依赖倒置原则）：高层模块不应该依赖低层模块，两者都应该依赖其抽象；抽象不应该依赖细节，细节应该依赖抽象。

#### **28\. 什么是“深拷贝”与“浅拷贝”？**

* **答案**：  
  * **浅拷贝**：只复制对象本身和对象内的基本数据类型变量，而对于对象内的引用类型变量，只复制了引用地址，新旧对象依然共享同一个子对象。  
  * **深拷贝**：不仅复制对象本身，还会递归地将对象内部持有的所有引用对象也完全复制一份，新旧对象彻底独立，互不影响。

#### **29\. 接口（Interface）和抽象类（Abstract Class）的区别？**

* **答案**：  
  * **设计意图**：抽象类是对本质的抽象，体现的是 “Is-A” 关系；接口是对行为的抽象，体现的是 “Like-A” 关系。  
  * **实现限制**：一个类只能继承一个抽象类（单继承）；但可以实现多个接口（多实现）。  
  * **成员变量/方法**：抽象类可以有非静态和非 final 的普通成员变量，可以有具体实现的方法；接口在旧版本中只能有静态常量（public static final）和抽象方法（Java 8 之后可以有 default 方法和 static 方法）。

#### **30\. 简述乐观锁和悲观锁的区别。**

* **答案**：  
  * **悲观锁**：总是假设最坏的情况，认为每次拿数据别人都会修改，所以在整个数据处理过程中将数据锁定。如 synchronized、ReentrantLock。  
  * **乐观锁**：总是假设最好的情况，认为每次拿数据别人不会修改，但在更新时会判断此期间别人有没有去更新这个数据（通过版本号机制或 CAS）。适合读多写少的场景。

#### **31\. 什么是 CAS？它存在什么问题？**

* **答案**：CAS（Compare And Swap，比较并交换）是一种无锁原子算法。它包含三个操作数：内存值 V、预期原值 A、新值 B。如果 V 的值等于 A，则将 V 的值设为 B，否则什么都不做。  
  **缺陷**：1. **ABA 问题**（值从 A 变成 B 又变成 A，CAS 无法察觉，可通过增加版本号解决，如 AtomicStampedReference）；2. **循环时间长开销大**（如果自旋长期不成功，对 CPU 带来大负荷）；3. **只能保证一个共享变量的原子操作**。

#### **32\. 简述接口中 default 关键字的作用（Java 8）。**

* **答案**：Java 8 允许在接口中加入具体的方法实现，但必须使用 default 关键字修饰。这样做的核心目的是为了**向后兼容**。当我们需要给一个历史悠久的现有接口添加新行为时，如果不用 default，所有实现该接口的类都必须被迫重写该新方法，而 default 方法可以让子类直接继承该默认实现，无需修改子类源码。

#### **33\. 解释 final、finally 和 finalize 的区别。**

* **答案**：  
  * final：修饰符。修饰类表示不可继承；修饰方法表示不可被重写；修饰变量表示常量，赋值后不可修改。  
  * finally：异常处理机制的一部分，通常与 try-catch 搭配使用，无论是否发生异常，finally 块中的代码都必定会执行（常用于资源关闭）。  
  * finalize：Object 类的一个方法。在垃圾回收器准备释放对象占用的内存空间前，会调用该方法，给予对象最后一次逃脱 GC 的机会（Java 9 开始已被废弃）。

#### **34\. 为什么 Java 中 String 是不可变的？**

* **答案**：1. **字符串常量池的需要**：只有当 String 不可变时，字符串池才能发挥作用，多个引用指向同一个常量可以大幅节省堆内存。2. **安全性**：String 经常作为网络连接、文件路径、数据库连接的参数，如果可变，极易被恶意篡改。3. **线程安全**：因为不可变，多个线程并发读取是绝对安全的。4. **Hash 值的唯一性**：String 缓存了自身的 hashCode，这使得它作为 HashMap 的 Key 时效率极高。

#### **35\. 简述 CountDownLatch 和 CyclicBarrier 的区别。**

* **答案**：  
  * CountDownLatch 是一个线程（或多个）等待其他 N 个线程完成某件事情后才能继续执行，计数器是一次性的，不能复用。  
  * CyclicBarrier 是 N 个线程之间互相等待，只有当所有线程都到达栅栏点（Barrier point）时，它们才能继续往下执行，计数器可以使用 reset() 方法重置并复用。

#### **36\. 什么是 Java 的线程生命周期？有哪些状态？**

* **答案**：Java 线程有 6 种状态：  
  1. NEW（新建状态）：创建后尚未启动的线程。  
  2. RUNNABLE（可运行状态）：包括正在运行或等待 CPU 分配时间片的线程。  
  3. BLOCKED（阻塞状态）：等待获取一个排他锁。  
  4. WAITING（等待状态）：无限期等待另一个线程执行特定操作（如调用了 Object.wait() 且无超时设定）。  
  5. TIMED\_WAITING（超时等待状态）：在指定时间内等待另一个线程的操作（如 Thread.sleep(ms)）。  
  6. TERMINATED（终止状态）：线程执行完毕已结束。

#### **37\. 类加载的生命周期包括哪些阶段？**

* **答案**：包括 5 个主要阶段：**加载（Loading）**、**连接（Linking，包含验证、准备、解析三个步骤）**、**初始化（Initialization）**、使用（Using）和卸载（Unloading）。

#### **38\. 在类加载的“准备”和“初始化”阶段，静态变量的赋值有什么不同？**

* **答案**：在**准备阶段**，JVM 只为类变量（static 变量）在方法区分配内存，并设置其初始默认零值（如 int 为 0，引用类型为 null）；而在**初始化阶段**，JVM 才会真正执行类构造器 \<clinit\> 方法，将类变量赋值为代码中实际指定的初始值（如 public static int a \= 123; 在准备阶段 a 为 0，在初始化阶段 a 变为 123）。如果是 static final 修饰的常量，在编译期就会放入常量池，准备阶段就会直接被赋值。

#### **39\. 强双亲委派模型可以被打破吗？如何打破？**

* **答案**：可以打破。主要有三种机制：  
  1. 自定义类加载器，重写 loadClass() 方法（如果重写 findClass() 依然符合双亲委派）。  
  2. 使用**线程上下文类加载器（Thread Context ClassLoader）**，如 SPI 机制（JDBC 驱动加载），允许父类加载器请求子类加载器去加载类。  
  3. 为了实现热插拔、热部署（如 OSGi、或者 Android 中的热修复/插件化技术），将双亲委派的树状结构改为了复杂的网状拓扑结构。

#### **40\. 简述快慢指针在单链表中的应用。**

* **答案**：快慢指针是指设置两个步长不同的指针（通常快指针每次走 2 步，慢指针每次走 1 步）。  
  应用场景：1. **判断单链表是否有环**（若有环，快慢指针必然在环内相遇）；2. **寻找链表的中点**（快指针走到链表末尾时，慢指针刚好在中点位置）；3. **寻找倒数第 K 个节点**（让快指针先走 K 步，然后两个指针同速前进，快指针到尾部时，慢指针即在目标位置）。

### **第二部分：计算机基础（网络与操作系统）（第 41-55 题）**

#### **41\. 简述 TCP 三次握手和四次挥手的过程及原因。**

* **答案**：  
  * **三次握手**：客户端发送 SYN，服务端返回 SYN+ACK，客户端再返回 ACK。**原因**：为了确认双方的接收能力和发送能力都是正常的，初始化序列号。  
  * **四次挥手**：主动方发 FIN，被动方回 ACK（此时进入半关闭，被动方可能继续发数据），被动方数据发完后发 FIN，主动方回 ACK 并等待 2MSL 后关闭。**原因**：因为 TCP 是全双工的，关闭时两个方向的连接必须单独关闭。

#### **42\. 为什么 TCP 四次挥手最后主动关闭方要等待 2MSL 才能彻底关闭？**

* **答案**：1. 为了保证最后一个 ACK 报文能够安全到达被动关闭方。如果这个 ACK 丢失，被动方会重发 FIN，处于 2MSL 等待期内的主动方可以接收并重新发送 ACK。2. 防止“已失效的连接请求报文段”出现在本连接中，让当前连接产生的所有报文段都在这段时间内从网络中消失。

#### **43\. HTTP 与 HTTPS 的区别？HTTPS 的加密原理是什么？**

* **答案**：HTTP 是明文传输，监听在 80 端口，不安全；HTTPS 是基于安全套接字层的 HTTP，数据加密传输，监听在 443 端口，需要 CA 证书。  
  **加密原理**：采用**混合加密**。在连接建立阶段（SSL 握手），使用**非对称加密**（公钥和私钥）来安全地传输对称加密的密钥；连接建立成功后，双方使用该**对称加密**密钥进行后续应用层数据的加密传输。

#### **44\. 简述 HTTP 状态码 301、302、401、403、500、503 的含义。**

* **答案**：  
  * 301：永久重定向（Moved Permanently）。  
  * 302：临时重定向（Found）。  
  * 401：未授权（Unauthorized），要求身份验证。  
  * 403：服务器拒绝请求（Forbidden），无访问权限。  
  * 500：服务器内部错误（Internal Server Error）。  
  * 503：服务器当前无法处理请求（Service Unavailable），通常是超载或停机维护。

#### **45\. HTTP 缓存机制中的强缓存和协商缓存有什么区别？**

* **答案**：  
  * **强缓存**：不向服务器发送请求，直接从本地缓存读取数据。通过响应头中的 Expires 或 Cache-Control（如 max-age）来判断是否命中。  
  * **协商缓存**：向服务器发送请求，由服务器判断本地缓存是否过期。如果没过期，服务器返回 304 Not Modified，客户端继续使用本地缓存。涉及的字段有 Last-Modified / If-Modified-Since 和 ETag / If-None-Match。

#### **46\. 什么是 GET 和 POST 的实质区别？**

* **答案**：根据 HTTP 规范，GET 用于获取资源，是**幂等**和**安全的**，参数拼接在 URL 后面，浏览器对 URL 长度有限制，且会被缓存留痕；POST 用于传输/修改资源，是非幂等的、不安全的，参数放在 Request Body 中，理论上无大小限制。

#### **47\. 什么是进程？什么是线程？它们有什么区别？**

* **答案**：**进程**是操作系统分配资源（如内存、CPU时间）的最小单位，拥有独立的地址空间。**线程**是 CPU 调度的最小单位，是进程内的一个执行流。  
  一个进程可以包含多个线程，多个线程共享该进程的堆和方法区资源，但每个线程拥有独立的程序计数器和栈空间。进程切换开销大，线程切换开销小。

#### **48\. 进程间通信（IPC）的方式有哪些？Android 独有的 IPC 是什么？**

* **答案**：传统 IPC 方式有：管道（Pipe）、信号（Signal）、消息队列（Message Queue）、共享内存（Shared Memory）、信号量（Semaphore）、套接字（Socket）。  
  Android 独有且最主要的 IPC 机制是 **Binder**，此外还常用 Socket（如和 Zygote 通信）、文件共享、ContentProvider。

#### **49\. 简述 Binder 机制的优势和底层原理。**

* **答案**：  
  * **优势**：1. **性能高**（传统 IPC 如管道/Socket 需拷贝 2 次数据，共享内存无需拷贝但同步复杂，而 Binder 仅需 **1 次拷贝**）；2. **安全性高**（Binder 机制在内核层为每个进程分配了 UID/PID，服务端可以安全地校验调用者的身份）。  
  * **底层原理**：基于内存映射（mmap）。在内核空间和接收方用户空间之间建立了一片物理内存的映射，发送方只需将数据从其用户空间拷贝到内核空间的这块映射区域，接收方就能直接读取，实现了单次拷贝。

#### **50\. 什么是死锁（操作系统层面）？如何检测和避免？**

* **答案**：指两个或多个进程在执行过程中，因争夺资源而造成的一种互相等待、若无外力作用都无法向前推进的僵局。  
  **避免/检测方法**：使用**银行家算法**进行动态资源分配流控制以避免进入不安全状态；使用资源分配图进行死锁检测；采用剥夺资源、撤销进程等方法进行死锁解除。

#### **51\. 简述操作系统中虚拟内存的作用。**

* **答案**：1. **扩大物理内存的边界**：允许程序使用比实际物理内存大得多的内存空间（利用磁盘 swap/页面置换）。2. **内存隔离与保护**：为每个进程提供独立的虚拟地址空间，防止进程之间相互破坏对方的数据。3. **简化内存管理**：将连续的虚拟内存映射到离散的物理内存中，减少了内存碎片的管理开销。

#### **52\. 什么是 TCP 粘包/拆包？怎么解决？**

* **答案**：TCP 是面向字节流的协议，它底层并不了解上层业务数据的具体边界，会根据网卡缓冲区和 MTU 大小将数据分块发送。这导致多个业务报文被合并为一个 TCP 包发送（粘包），或者一个业务报文被拆分为多个 TCP 包发送（拆包）。  
  **解决方法**：1. **定长协议**（每个报文固定长度）；2. **分隔符协议**（在报文尾部加特定特殊字符分割）；3. **定义消息头加消息体**（在消息头中携带一个字段表示整个消息体的总长度）。

#### **53\. 解释 DNS 的工作原理。**

* **答案**：DNS（域名系统）负责将人类易读的域名（如 baidu.com）解析为计算机识别的 IP 地址。  
  **流程**：当客户端发起请求时，首先查本地 hosts 文件和浏览器缓存 \-\> 未命中则请求**本地 DNS 服务器（递归查询）** \-\> 本地 DNS 依次请求根域名服务器、顶级域名服务器（.com）、权限域名服务器（迭代查询），最终将获取到的 IP 地址返回给客户端并缓存。

#### **54\. 计算机网络中 OSI 七层模型和 TCP/IP 四层模型分别是什么？**

* **答案**：  
  * **OSI 七层模型**：物理层、数据链路层、网络层、传输层、会话层、表示层、应用层。  
  * **TCP/IP 四层模型**：网络接口层、网络层（IP）、传输层（TCP/UDP）、应用层（HTTP/FTP/DNS等）。

#### **55\. 什么是应用层协议 HTTP/2 的多路复用？**

* **答案**：在 HTTP/1.x 中，浏览器针对同一个域名的并发连接数有限制（通常为 6 个），且存在队头阻塞问题。HTTP/2 引入了**二进制分帧层**，将所有的传输信息分割为更小的帧。它允许在**同一个 TCP 连接**上同时交错发起无数个请求和响应，多个请求的帧不会互相阻塞，彻底解决了应用层的队头阻塞，极大地提高了带宽利用率。

### **第三部分：Android 基础面试题（第 56-80 题）**

#### **56\. 请述 Activity 的生命周期。在 Activity A 跳转到 Activity B 时，两者的生命周期如何回调？**

* **答案**：Activity 的基本生命周期包括：onCreate \-\> onStart \-\> onResume \-\> onPause \-\> onStop \-\> onDestroy，以及被覆盖/弹窗时的 onRestart。  
  当 A 跳转到 B 时，回调顺序为：**A.onPause() \-\> B.onCreate() \-\> B.onStart() \-\> B.onResume() \-\> A.onStop()**。注意：只有当 A 的 onPause 执行完，B 才会开始创建，因此 onPause 中切忌做耗时操作。

#### **57\. Activity 的 4 种启动模式（LaunchMode）分别是什么？**

* **答案**：  
  * standard（标准模式）：每次启动都会创建一个新的 Activity 实例放入调用者所在的栈顶。  
  * singleTop（栈顶复用模式）：如果新 Activity 已经位于任务栈的栈顶，则不会重新创建，而是复用它并回调其 onNewIntent 方法。  
  * singleTask（栈内复用模式）：单例模式。如果栈内存在该 Activity 实例，则将其上的所有其他 Activity 全部出栈（Clear Top 功能），使其移至栈顶，并回调 onNewIntent。  
  * singleInstance（单实例模式）：加强版 singleTask。该 Activity 会独自占用一个新的任务栈，且整个系统中该 Activity 只有一个实例。

#### **58\. 简述 Fragment 的生命周期，它与 Activity 的生命周期有什么关系？**

* **答案**：Fragment 拥有独特的生命周期：onAttach \-\> onCreate \-\> onCreateView \-\> onActivityCreated \-\> onStart \-\> onResume \-\> onPause \-\> onStop \-\> onDestroyView \-\> onDestroy \-\> onDetach。  
  Fragment 必须嵌入在 Activity 中，其生命周期紧密受宿主 Activity 的控制。例如，当 Activity 变为 onStart 时，其内部绑定的 Fragment 才会进入 onStart 阶段。

#### **59\. 简述 Service 的两种启动方式及区别。**

* **答案**：  
  * startService()：一旦启动，Service 与调用者的生命周期无关，即使调用者 Activity 被销毁，Service 依然会在后台长期运行。通过 stopService() 或 Service 内部调用 stopSelf() 来停止。  
  * bindService()：调用者与 Service 形成绑定绑定关系，生命周期绑定。调用者销毁时，Service 会自动解绑并销毁。通过 unbindService() 主动解绑。可以通过 Binder 接口实现与 Service 的通信。

#### **60\. 什么是 IntentService？它与普通 Service 的区别？**

* **答案**：普通 Service 默认运行在**主线程**中，不能直接进行耗时操作。IntentService 是 Service 的子类，它内部维护了一个 HandlerThread（串行工作线程）。  
  每次通过 startService 提交的任务都会被放入工作线程的队列中依次异步处理，处理完毕后 IntentService 会**自动调用 stopSelf() 关闭自己**，无需手动管理，适合执行后台串行的单次耗时任务。

#### **61\. 简述 Android 消息机制（Handler、Looper、MessageQueue）的工作原理。**

* **答案**：Handler 机制由四部分组成：  
  * Message：消息载体。  
  * MessageQueue：消息队列，底层是单链表，按时间顺序排列消息。  
  * Looper：消息循环器。每个线程最多持有一个 Looper，它内部包含一个死循环（Looper.loop()），不断地从 MessageQueue 中取出消息（没有消息时通过 Linux 的 epoll 机制阻塞等待）。  
  * Handler：消息的发送者与处理者。通过 sendMessage 将消息压入 MessageQueue，Looper 取出消息后，交由该消息关联的 handler.dispatchMessage 回调处理。

#### **62\. 为什么子线程中不能直接 new Handler()？如何正确在子线程创建 Handler？**

* **答案**：因为 Handler 的创建依赖于当前线程的 Looper 对象。子线程默认是没有初始化 Looper 的，直接 new 会抛出 "Can't create handler inside thread that has not called Looper.prepare()" 异常。  
  **正确做法**：先调用 Looper.prepare() 为当前线程初始化 Looper，然后创建 Handler，最后调用 Looper.loop() 开启消息循环。或者直接使用官方提供的 HandlerThread。

#### **63\. Looper.loop() 是一个死循环，为什么不会导致应用卡死（ANR）？**

* **答案**：应用卡死（ANR）是指主线程在处理某个消息或事件时耗时过长，导致无法及时响应用户的输入事件。  
  而 Looper.loop() 的死循环是 Android app 的主驱动运行方式，所有 Activity 生命周期、UI 刷新事件本质上都是作为一个个 Message 扔给这个死循环来执行的。当 MessageQueue 中没有消息时，主线程会通过 Linux 的 epoll\_wait 挂起，释放 CPU 资源进入睡眠状态；当有新消息进来（如触摸事件）时，会通过文件描述符写入从而唤醒主线程。这种“没活干就睡觉，有活干就工作”的机制不会造成系统卡死。

#### **64\. 什么是 ANR？如何规避和分析 ANR？**

* **答案**：ANR（Application Not Responding，应用无响应）。主线程在特定时间内未处理完核心事件就会触发：输入事件响应超时（5s）、BroadcastReceiver 耗时（前台 10s，后台 60s）、Service 耗时（前台 20s，后台 200s）。  
  **规避**：绝对不在主线程做任何耗时操作（如网络、I/O、大计算），将其异步化。  
  **分析**：通过提取并分析系统生成的 /data/anr/traces.txt 文件，查看主线程的调用栈，定位是在哪个方法上发生了阻塞。

#### **65\. 简述 Android 的四大组件及其作用。**

* **答案**：  
  * Activity：展现给用户的视觉界面，负责与用户进行直接交互。  
  * Service：无界面，在后台执行长期运行任务的组件。  
  * BroadcastReceiver：广播接收器，用于接收并响应来自系统或其他应用发布的全局广播消息。  
  * ContentProvider：内容提供者，用于在不同的应用之间安全地共享和管理数据。

#### **66\. 简述 Serializable 和 Parcelable 的区别，Android 中应该如何选择？**

* **答案**：  
  * Serializable：Java 原生提供的序列化接口，基于空标记。它在序列化时涉及大量的 **I/O 操作**，并使用**反射**机制产生大量的临时对象，效率较低，但使用极其简单。通常用于磁盘存储或网络传输。  
  * Parcelable：Android 独有的高效序列化接口。它将对象直接在**内存**中进行拆分和组装，效率极高，没有反射开销。通常用于 Android 进程间通信（IPC）、Intent 传值。缺点是编写代码稍显冗长。

#### **67\. 什么是 Context？它的继承体系是怎样的？**

* **答案**：Context（上下文）是 Android 系统的核心桥梁，它提供了访问系统资源（如 AssetManager）、启动组件、获取系统服务等核心能力。  
  **继承体系**：Context 是一个抽象类。它的直接子类是 ContextWrapper。而 Application、Service 以及 ContextThemeWrapper（Activity 的父类）都继承自 ContextWrapper。

#### **68\. Application Context 与 Activity Context 的区别？**

* **答案**：  
  * Application Context：生命周期贯穿整个应用的生命周期。它不依赖于任何特定的 UI，适合用于单例、工具类等需要长期持有 Context 且与界面无关的场景。  
  * Activity Context：生命周期与特定的 Activity 绑定。它包含有特定的 UI 主题（Theme）。**不能**在长生命周期的对象中持有它（否则会导致 Activity 内存泄漏），启动 Activity、弹出 Dialog 等涉及 UI 展现的操作必须使用 Activity Context。

#### **69\. 简述 ListView 与 RecyclerView 的核心区别。**

* **答案**：RecyclerView 是 ListView 的升级替代版：  
  1. **布局管理器**：RecyclerView 通过 LayoutManager 统一支持线性、网格、瀑布流等丰富布局， ListView 仅支持垂直线性。  
  2. **复用机制**：ListView 是两级缓存，RecyclerView 拥有更强大的**四级缓存机制（Scrap、CacheView、ViewCacheExtension、RecycledViewPool）**。  
  3. **ViewHolder**：ListView 的 ViewHolder 是选配（靠开发者自己写），RecyclerView 强制要求使用 ViewHolder 模式。  
  4. **局部刷新**：RecyclerView 支持针对单项数据发生改变时的局部刷新（notifyItemChanged），ListView 只能重绘整个列表。  
  5. **动画**：RecyclerView 内置了条目添加、删除的精美过渡动画，ListView 很难实现。

#### **70\. 什么是 View 树的“层级扁平化”？为什么要扁平化？**

* **答案**：层级扁平化是指通过合理选择布局控件（如使用 ConstraintLayout 代替多层嵌套的 LinearLayout 或 RelativeLayout），从而减少 View 树的深度（嵌套层级）。  
  **原因**：View 在进行测量（Measure）和布局（Layout）时，多层嵌套会导致严重的递归调用。尤其是部分布局（如 RelativeLayout 或设置了 weight 的 LinearLayout）会触发**二次测量**。View 树越深，整体测量耗时呈指数级增长，极易导致掉帧卡顿。

#### **71\. 简述 Android 的事件分发机制。**

* **答案**：事件分发主要涉及三个核心方法：dispatchTouchEvent（分发）、onInterceptTouchEvent（ViewGroup独有，拦截）、onTouchEvent（消费）。  
  当点击事件发生时，事件由系统的 Activity 逐层向下传递：**Activity \-\> PhoneWindow \-\> DecorView \-\> 各级 ViewGroup \-\> Target View**。在传递过程中，ViewGroup 可以通过 onInterceptTouchEvent 拦截事件。若没有被拦截，事件最终到达 Target View。如果 Target View 也不消费该事件（onTouchEvent 返回 false），事件将顺着原路逆向向上回传，最终交由 Activity 处理。

#### **72\. 怎么解决 View 的滑动冲突？**

* **答案**：主要有两种经典解决方法：  
  * **外部拦截法（推荐）**：由父容器决定是否拦截事件。在父容器的 onInterceptTouchEvent 中，根据当前滑动的距离、方向等业务逻辑（例如横向滑动还是纵向滑动）决定是否拦截。如果需要拦截则返回 true，后续事件就直接交给父容器的 onTouchEvent 处理。  
  * **内部拦截法**：由子 View 控制是否允许父容器拦截。子 View 通过调用 getParent().requestDisallowInterceptTouchEvent(true/false) 来干预父容器的行为。当子 View 需要处理事件时，禁止父容器拦截；反之则放开拦截权限。

#### **73\. 简述 View 的绘制流程。**

* **答案**：View 的整个绘制流程由 ViewRootImpl 触发，顺着 View 树依次执行三个核心阶段：  
  1. **Measure（测量）**：确定 View 的具体原始大小。通过测量规格 MeasureSpec（由父布局的 Spec 和自身的 LayoutParams 共同决定）递归测量每个子 View。  
  2. **Layout（布局）**：确定 View 在父容器中的四个顶点的具体坐标。  
  3. **Draw（绘制）**：将 View 绘制到屏幕上。包含绘制背景、绘制自己（onDraw）、绘制子 View（dispatchDraw）、绘制装饰（如滚动条）等步骤。

#### **74\. 什么是 MeasureSpec？它有哪三种模式？**

* **答案**：MeasureSpec 是一个 32 位的 int 值，高 2 位代表**测量模式（SpecMode）**，低 30 位代表**测量大小（SpecSize）**。  
  三种模式：  
  * EXACTLY：精确模式。当控件的 layout\_width 或 layout\_height 指定为具体数值（如 100dp）或 match\_parent 时触发。  
  * AT\_MOST：最大值模式。对应 wrap\_content，子 View 的大小不能超过父容器允许的最大剩余空间。  
  * UNSPECIFIED：未指定模式。父容器不对子 View 做任何大小限制，通常用于系统内部、ListView 或 RecyclerView 内部。

#### **75\. 简述 Android 广播（Broadcast）的两种注册方式及区别。**

* **答案**：  
  * **静态注册**：在 AndroidManifest.xml 中配置。其特点是即使应用处于关闭未启动状态，只要系统发布了匹配的广播，应用也会被自动唤醒并接收。适合接收一些系统全局通知（如开机广播）。  
  * **动态注册**：在代码中通过 context.registerReceiver() 显式注册。它的生命周期严格依附于绑定的组件（如 Activity），在组件销毁时**必须**调用 unregisterReceiver() 解绑，否则会导致严重的内存泄漏。其安全性和灵活性高于静态注册。

#### **76\. 什么是本地广播（LocalBroadcastManager）？为什么要使用它？**

* **答案**：普通的系统广播属于全局广播，应用发出的广播可以被全系统任何其他应用截获，同样应用也会被动接收外来的恶意广播，存在安全隐患和效率问题。  
  LocalBroadcastManager 机制发出的广播**只能在应用内部传播**。其底层本质上并不是通过 Binder 的 IPC 机制和系统的 AMS 通信，而是采用应用内部的 **Handler 机制**进行消息分发。它不仅极其安全，而且由于没有进程间通信开销，效率非常高。（注：Jetpack 时代更推荐使用 LiveData/Flow 替代它）。

#### **77\. 简述 ContentProvider 的作用及跨进程数据共享的原理。**

* **答案**：ContentProvider 是四大组件之一，主要用于跨进程封装底层的各种数据源（如 SQLite 数据库、文件等），向其他应用提供统一、标准的安全增删改查（CRUD）接口。  
  **原理**：其底层通信完全基于 **Binder IPC**。当其他应用通过 ContentResolver 发起请求时，系统会通过 Binder 建立连接，并将数据以 CursorWindow 的形式进行共享。CursorWindow 内部利用了匿名共享内存（Ashmem）技术，使得跨进程传输大批量结构化数据时速度极快。

#### **78\. 什么是 SurfaceView？它与普通 View 的区别是什么？**

* **答案**：普通 View 的绘制都在主线程（UI 线程）中进行，并且和宿主窗口共享同一个绘图表面。如果绘制逻辑太复杂（如游戏、视频播放），会导致主线程阻塞卡顿。  
  SurfaceView 继承自 View，但它内部拥有一个**独立的物理绘图表面（Surface）**，并且允许开发者开辟一个**子线程**在其独立的 Surface 上进行异步的完全自绘像素操作。它绕过了复杂的 UI 视图树树重绘机制，非常适合处理高帧率、重度动画及音视频渲染的场景。

#### **79\. 简述 Android 动画的三种分类。**

* **答案**：  
  * **补间动画（Tween Animation）**：对 View 进行旋转、平移、缩放、透明度变化。它**只改变了 View 的视觉绘制效果，并没有真正改变 View 的属性坐标**（例如平移后的 View 其点击事件还在原位）。  
  * **逐帧动画（Frame Animation）**：将一系列图片按顺序快速播放，类似放电影。如果图片大而多，极易引发 OOM。  
  * **属性动画（Property Animation \- Java 3.0引入）**：通过动态改变对象的真实属性（如 translationX、alpha）来实现动画。它不仅可以作用于 View，还能作用于任意 Java 对象，动画停止后对象的实际物理属性确实发生了改变。

#### **80\. 什么是 Window、PhoneWindow 和 DecorView 的关系？**

* **答案**：  
  * Window 是一个最高层的抽象视窗类，定义了窗口的一切视觉和控制行为规范。  
  * PhoneWindow 是 Window 的**唯一具体实现类**。每个 Activity 内部都持有一个 PhoneWindow 实例。  
  * DecorView 是 PhoneWindow 内部的一个顶层根布局（继承自 Framelayout），它包含了系统的状态栏、标题栏以及用户在 setContentView 中传入的布局。所有 Activity 的画面本质上都是绘制在 DecorView 上的。

### **第四部分：Android 高级进阶、性能优化与设计模式（第 81-100 题）**

#### **81\. 什么是图片加载框架（如 Glide）中的三级缓存？**

* **答案**：三级缓存是经典的图片加载网络优化策略：  
  1. **内存缓存（第一级）**：优先从内存中读取图片（Glide 中分为活跃资源和 LRU 内存缓存）。速度最快，没有 I/O 开销。  
  2. **磁盘/本地缓存（第二级）**：若内存中没有，则去 App 的本地沙盒或 SD 卡中查找之前下载好的缓存文件。  
  3. **网络下载（第三级）**：若本地也没有，则发起 HTTP 网络请求从服务器下载图片，下载成功后同步写入磁盘缓存和内存缓存中，以便下次使用。

#### **82\. 简述 Android 内存优化中针对图片（Bitmap）的优化策略。**

* **答案**：Bitmap 是 Android 中的“内存大户”，常用优化策略包括：  
  1. **图片等比缩放**：利用 BitmapFactory.Options 里的 inJustDecodeBounds 预读图片尺寸，计算合理的 inSampleSize 采样率进行压缩加载。  
  2. **像素格式优化**：根据需求将默认的 ARGB\_8888（每个像素4字节）改为 RGB\_565（每个像素2字节，无透明通道），直接省下一半内存。  
  3. **Bitmap 复用**：利用 inBitmap 属性，复用已存在且大小合适的等长内存块，减少内存的频繁申请与销毁，规避**内存抖动**。  
  4. **及时回收**：不用的大图及时调用 bitmap.recycle()（在旧版本中尤为重要），避免长生命周期对象持有。

#### **83\. 什么是内存抖动？为什么会导致卡顿甚至 OOM？**

* **答案**：内存抖动是指在极短的时间内，程序频繁地创建了大量的临时对象，随后又快速地将其释放（例如在循环体或 onDraw 方法中 new 对象）。  
  这会导致内存占用呈锯齿状剧烈波动。此时 JVM 的垃圾回收器（GC）会被迫高频度地触发。GC 运行时会引发 **Stop The World (STW)** 现象，即暂停主线程的一切工作，从而导致界面出现严重的掉帧、卡顿；高频的内存碎片积压最终甚至会导致系统抛出 OOM。

#### **84\. 简述 经典的架构模式 MVC、MVP 和 MVVM 的本质区别。**

* **答案**：  
  * **MVC**：View 接收用户操作传给 Controller，Controller 修改 Model，Model 改变后**直接通知 View** 更新。各层之间交叉依赖，导致 Activity 往往充当了 Controller 又充当了 View，代码臃肿。  
  * **MVP**：切断了 View 与 Model 的直接联系。通过引入 **Presenter（主持人）**，View 和 Presenter 之间通过**定义明确的双向接口契约**进行通信。虽然解耦彻底，但会导致产生海量的接口文件，且 Presenter 容易持有 View 产生内存泄漏。  
  * **MVVM**：核心是 **ViewModel**。采用**数据双向绑定（Data Binding）或响应式状态流（LiveData/Flow）**。当 Model 的数据改变时，View 会通过观察者机制**自动双向更新**，无需开发者手动编写繁琐的 UI 刷新代码（如 setText），实现了真正的声明式数据驱动。

#### **85\. 简述 Jetpack 组件中的 Lifecycle 的核心作用和原理。**

* **答案**：  
  * **作用**：Lifecycle 允许普通的第三方非组件类（如视频播放器、地理位置定位器）能够**直接感应并观察** Activity 或 Fragment 的生命周期变化，从而在类内部自行处理 onStart、onStop 的资源初始化与释放，避免了在 Activity 中编写大量臃肿的样板回调代码。  
  * **原理**：通过在组件（如 ComponentActivity）内部嵌入一个无界面的 ReportFragment。通过这个 Fragment 拦截并感知宿主的生命周期变化，再借助 LifecycleRegistry（状态机管理类）将生命周期事件分发给所有注册的观察者（LifecycleObserver）。

#### **86\. 什么是 LiveData？它有什么独特的优势？**

* **答案**：LiveData 是一种可被观察的数据持有者类，专门为 Jetpack 架构打造。  
  **优势**：1. **感应生命周期**：它能严格确保只在观察者（如 Activity）处于活跃状态（STARTED 或 RESUMED）下才向其推送最新数据，非活跃状态自动暂停推送。2. **防止内存泄漏**：当观察者的生命周期进入 DESTROYED 状态时，LiveData 会自动自我注销观察者，彻底杜绝内存泄漏隐患。3. **杜绝因 Activity 销毁导致的崩溃**：如果 Activity 已经结束，它绝不会收到任何异步回传的数据，避免了执行 UI 赋值带来的空指针等风险。

#### **87\. 简述 ViewModel 的生命周期特性及原理。**

* **答案**：ViewModel 的生命周期是**独立且长于** Activity 的。当 Activity 因为配置发生改变（如屏幕旋转、语言切换）而被系统销毁重建时，ViewModel 实例在内存中会被**完好地保留下来**，重新绑定到新的 Activity 实例上。只有当 Activity 真正彻底完结（如按下返回键销毁）时，ViewModel 才会触发 onCleared() 释放。  
  **原理**：Activity 内部持有一个 ViewModelStore。在因配置改变触发重建时，系统会通过特殊的非销毁机制（onRetainNonConfigurationInstance）将该存储库保留在内存中，从而实现了跨 Activity 实例的数据共享。

#### **88\. 什么是 Android 中的 APK 瘦身（减小应用体积）？有哪些核心手段？**

* **答案**：APK 瘦身是指在保证功能完整的前提下，最大化压缩安装包大小：  
  1. **代码混淆**：启用 ProGuard/R8，进行无用代码裁剪（Tree Shaking）和类名方法名转义短字符。  
  2. **资源优化**：利用 shrinkResources 自动剔除无用资源；全面将大图转换为 WebP 格式，或使用 VectorDrawable（矢量图）代替多套分辨率位图。  
  3. **动态化与 App Bundle**：发布采用 Google Play 的 AAB 格式，让用户仅下载对应其手机架构和屏幕密度的拆分资源包。  
  4. **SO 库裁剪**：通常只保留主流的 armeabi-v7a 或 arm64-v8a，剔除不常用的 x86 架构 SO 文件。

#### **89\. 简述 观察者模式（Observer Pattern）在 Android 中的代表。**

* **答案**：定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都会得到通知并自动更新。  
  **Android代表**：RxJava、LiveData、Kotlin Flow、以及各种平台的各种基础控件 Listener 监听注册机制。

#### **90\. 什么是策略模式（Strategy Pattern）？**

* **答案**：定义一系列可相互替换的算法，并把它们一个个独立封装起来，使得算法可以独立于使用它的客户端而变化。  
  **例如**：在 Android 开发中，为图片加载框架或网络请求框架设计一套统一的抽象代理接口，底层可以随意在 Glide、Coil 或 OkHttp、Volley 之间进行自由切换，而上层业务层代码不需要做任何改动。

#### **91\. 什么是单例模式？请手写一个线程安全的懒汉式单例。**

* **答案**：单例模式保证一个类仅有一个实例，并提供一个访问它的全局访问点。  
  Java  
  public class Singleton {  
      // 必须加 volatile 关键字防止指令重排序  
      private static volatile Singleton instance;

      private Singleton() {} // 私有构造函数

      public static Singleton getInstance() {  
          if (instance \== null) { // 第一次检查：规避不必要的同步锁开销  
              synchronized (Singleton.class) {  
                  if (instance \== null) { // 第二次检查：确保在多线程并发下依然单例  
                      instance \= new Singleton();  
                  }  
              }  
          }  
          return instance;  
      }  
  }

#### **92\. 目前主流的三大 UI 框架 Flutter、Jetpack Compose、React 的根本区别体现在哪里？**

* **答案**：体现在底层的**编程语言机制、渲染引擎和内存模型**完全不同：  
  * **Flutter**：使用 Dart 语言（AOT 编译），拥有完全独立的**自绘像素渲染引擎**（通过 Skia/Impeller），直接绕过原生平台的系统控件，自己向屏幕 Canvas 绘图。内存中长期驻留经典的 OOP 对象树（Element、RenderObject树）。  
  * **Jetpack Compose**：基于 Kotlin 语言，属于**半自绘/半原生**。它利用高度定制的**编译器插件状态机**，在 Android 平台上直接调用硬件加速 Canvas 进行极致的扁平化绘图。内存模型采用极简的线性数据结构 **Slot Table**。  
  * **React**：基于 JavaScript/TypeScript（JIT编译），运行在 Web 宿主浏览器环境。其最终渲染是通过浏览器内核将**虚拟 DOM 树**翻译为原生的 DOM 节点。采用双缓存 **Fiber 链表树**进行并发任务调度和 Diff 最小化更新。

#### **93\. 什么是 Android 插件化？其核心技术点是什么？**

* **答案**：插件化是指将一个大型 App 拆分为一个宿主和多个独立的插件 APK。在应用运行时，宿主可以动态地从未安装的插件 APK 中加载并运行其中的组件。  
  **核心技术**：1. **类加载器钩子**（通过自定义 DexClassLoader，将插件的 dex 文件动态合并注入到宿主的 BaseDexClassLoader 中）。2. **四大组件的“偷天换日”**（通过 Hook 系统的 AMS 或 Handler 机制，在清单文件中埋下一个合法的“占坑 Activity”，在系统校验时使用占坑 Activity 欺骗系统，在真正启动创建时再将其替换为插件中未注册的真实 Activity）。

#### **94\. 什么是 Android 热修复（HotFix）？它和插件化有什么区别？**

* **答案**：  
  * **热修复**：用于线上紧急 Bug 的无感知修复。通常是将修复好 Bug 的特殊补丁 dex 文件下发，利用类加载器的“插队”原理（让修复后的类排在 dexElements 数组的最前面，根据类加载的**缓存优先原则**，系统会优先加载修复后的类，从而让有 Bug 的旧类永远无法被执行到）。  
  * **区别**：热修复的目的是**修 Bug**，不需要增加新组件，体积小；插件化的目的是**功能动态扩展**，动态下发的是一整套庞大的新业务（包含大量未注册的 Activity 界面）。

#### **95\. 简述 进程保活（KeepAlive）在 Android 高版本中的演变。**

* **答案**：在旧版本中，开发者常通过一像素 Activity 监听锁屏、后台无声音乐循环播放、双进程守护（互相拉活）等流氓手段强行保活。  
  但在 Android 高版本（Android 8.0+）中，系统引入了严厉的后台执行限制（Background Execution Limits）、低电耗模式（Doze Mode）以及 OOM Killer 机制，上述流氓手段已全部失效。目前官方和主流认可的唯一保活路径是：**合理转换为前台服务（Foreground Service，必须展示通知栏）**、接入系统的**厂商推送（JobScheduler/WorkManager、FCM等）**，以及争取白名单。

#### **96\. 什么是 OkHttp 拦截器（Interceptor）？它采用了什么设计模式？**

* **答案**：拦截器是 OkHttp 的核心灵魂，负责对网络请求（Request）和网络响应（Response）进行层层转换、重试、缓存处理和桥接。  
  它采用了经典的**责任链模式（Chain of Responsibility）**。所有的拦截器（包括自定义拦截器、重定向拦截器、桥接拦截器、缓存拦截器、连接拦截器、网络呼叫拦截器）连成一条链，请求顺着链条一层层传进去封装，响应逆着链条一层层吐出来，设计极其优雅。

#### **97\. 什么是 内存溢出（OOM）？如何用工具定位和排查？**

* **答案**：OOM（OutOfMemoryError）是指应用申请的内存超过了系统或分配给该进程的虚拟内存最大上限，导致无法分配新内存而崩溃。  
  **定位排查**：1. 在开发阶段，集成官方推荐的 **LeakCanary** 库，它能自动化监控并直接输出完整的内存泄漏引用链。2. 使用 Android Studio 自带的 **Profilers (Memory Profiler)** 捕获并导出 **Hprof** 堆转储文件。3. 将 Hprof 文件导入到 专业工具 **MAT (Memory Analyzer Tool)** 中，通过查看 Histogram 和 Dominator Tree，分析大对象的引用路径（Incoming/Outgoing References），找出是谁在强引用大对象导致其无法被 GC 释放。

#### **98\. 简述 依赖注入框架（如 Dagger2 / Hilt）的优势及核心思想。**

* **答案**：依赖注入（Dependency Injection，DI）的核心思想是**控制反转（IoC）**。即一个类（如 Activity）内部所需要的其他复杂对象（如 NetworkService、DatabaseHelper），不再由该类自己去内部手动 new 出来，而是由外部框架统一创建并“注入”进来。  
  **优势**：彻底实现了解耦。使得代码的单元测试变得非常容易（可以轻松 mock 注入假对象）；极大提升了大型团队协作时组件的模块化和复用能力。

#### **99\. 什么是 Android 布局加载（LayoutInflater）的异步加载？**

* **答案**：默认情况下，LayoutInflater.inflate() 是在主线程执行的，它需要同步读取并解析 XML 的文本结构，然后再通过反射递归创建每个 View 对象。当布局非常庞大时，会造成严重的主线程耗时和掉帧。  
  Jetpack 提供了 AsyncLayoutInflater 库，它允许开发者在**子线程**中提前异步解析 XML 布局并生成 View 树，当子线程解析完毕后通过回调主线程将 View 树直接塞入容器中，从而消除了主线程反射创建布局的性能卡顿。

#### **100\. 简述 Kotlin 协程（Coroutines）中的“非阻塞式挂起”究竟怎么理解？**

* **答案**：非阻塞式挂起是协程的最大卖点。在传统多线程中，如果想等待一个耗时的网络结果，必须阻塞（Block）当前线程，这会导致 CPU 线程无法执行其他任务。  
  而 Kotlin 协程在遇到挂起函数（suspend）时，会将当前协程的执行状态“切走”（通过编译期生成的状态机），把耗时任务扔给工作线程处理，而**当前的主线程则被完全释放出来**，可以继续去绘制 UI、响应用户点击（即非阻塞）。当异步结果返回时，协程框架会自动发消息切回主线程，在刚才停下的地方“恢复”（Resume）执行。其底层本质上依然是基于 **Handler 的异步回调**，但让开发者可以用写**同步结构化代码**的方式去写异步逻辑。
