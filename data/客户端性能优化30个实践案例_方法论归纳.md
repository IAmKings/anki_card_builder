### 案例 1：冷启动优化 — Application 初始化任务分级调度

**问题背景**：某电商App冷启动耗时3.2秒，Application中初始化20+SDK，主线程阻塞严重。

**优化方案**：

```kotlin
// 1. 任务分级分类
sealed class InitTask(val name: String, val priority: Priority) {
    enum class Priority { CRITICAL, HIGH, NORMAL, LAZY }
    abstract fun execute()
}

// 2. 任务调度器
class StartupTaskManager {
    private val criticalTasks = mutableListOf<InitTask>()
    private val asyncTasks = mutableListOf<InitTask>()
    private val lazyTasks = mutableListOf<InitTask>()

    fun addTask(task: InitTask) {
        when (task.priority) {
            InitTask.Priority.CRITICAL -> criticalTasks.add(task)
            InitTask.Priority.HIGH -> asyncTasks.add(task)
            InitTask.Priority.NORMAL -> asyncTasks.add(task)
            InitTask.Priority.LAZY -> lazyTasks.add(task)
        }
    }

    fun execute() {
        criticalTasks.forEach { it.execute() }
        val latch = CountDownLatch(asyncTasks.size)
        asyncTasks.forEach { task ->
            Executors.newSingleThreadExecutor().execute {
                task.execute()
                latch.countDown()
            }
        }
        latch.await(2, TimeUnit.SECONDS)
        lazyTasks.forEach { task ->
            Looper.myQueue().addIdleHandler {
                task.execute()
                false
            }
        }
    }
}
```

**优化效果**：冷启动时间从3.2s降至1.4s，首帧时间减少60%。

**方法论归纳**：
> **"分级调度法"**：将初始化任务按重要性分为四级（Critical/High/Normal/Lazy），Critical主线程同步执行，High/Normal子线程并行，Lazy利用IdleHandler延迟执行。核心是"让首帧渲染不受非关键任务阻塞"。

### 案例 2：首屏预加载 — 利用 Idle 时间预请求数据

**问题背景**：首页进入后才开始请求Banner和商品列表数据，用户感知加载慢。

**优化方案**：

```kotlin
class PreloadManager(private val repository: HomeRepository) {
    private val preloadScope = CoroutineScope(SupervisorJob() + Dispatchers.IO)
    private val preloadCache = ConcurrentHashMap<String, Any>()

    fun preloadHomeData() {
        preloadScope.launch {
            val bannerDeferred = async { repository.fetchBanners() }
            val categoryDeferred = async { repository.fetchCategories() }
            val hotDeferred = async { repository.fetchHotItems() }
            preloadCache["banners"] = bannerDeferred.await()
            preloadCache["categories"] = categoryDeferred.await()
            preloadCache["hotItems"] = hotDeferred.await()
        }
    }

    fun <T> getPreloadedData(key: String): T? {
        @Suppress("UNCHECKED_CAST")
        return preloadCache.remove(key) as? T
    }
}
```

**优化效果**：首页数据加载时间从800ms降至100ms（缓存命中时）。

**方法论归纳**：
> **"时空置换法"**：利用用户无感知的空闲时间预加载后续所需数据，将网络请求的"时间成本"前置到"空间成本"（内存缓存）。核心是"在用户需要前，数据已经准备好"。

### 案例 7：自定义 View 的过度绘制优化

**问题背景**：复杂自定义View（如股票K线图）出现多层重叠绘制，GPU过度绘制严重。

**优化方案**：

```kotlin
class CandlestickView @JvmOverloads constructor(
    context: Context, attrs: AttributeSet? = null
) : View(context, attrs) {

    override fun onDraw(canvas: Canvas) {
        super.onDraw(canvas)

        canvas.save()
        canvas.clipRect(0f, 0f, width.toFloat(), height.toFloat())
        drawGrid(canvas)
        canvas.restore()

        canvas.save()
        canvas.clipRect(contentRect)
        candles.forEach { candle ->
            if (!candle.rect.intersects(contentRect)) return@forEach
            drawCandle(canvas, candle)
        }
        canvas.restore()
    }

    init { setLayerType(LAYER_TYPE_HARDWARE, null) }

    fun updateLastCandle(candle: Candle) {
        candles[candles.size - 1] = candle
        invalidate(candles.last().rect.toRect())  // 局部刷新

}
```

### 案例 8：Compose 重组优化 — 避免不必要的重组

**问题背景**：Compose列表滑动时频繁重组，导致掉帧。

**优化方案**：

```kotlin
@Immutable
data class ProductList(val products: ImmutableList<Product>)

@Composable
fun ProductList(products: List<Product>, query: String) {
    val filtered by remember(products, query) {
        derivedStateOf {
            if (query.isEmpty()) products
            else products.filter { it.name.contains(query, ignoreCase = true) }
        }
    }

    LazyColumn {
        items(filtered, key = { it.id }) { product ->
            ProductItem(product = product)
        }
    }
}

@Composable
fun ProductItem(product: Product) {
    key(product.id) {
        Column {
            Text(product.name)
            Text("${product.price}")
        }
    }
}
```

**优化效果**：重组次数减少80%，FPS稳定在55+。

**方法论归纳**：
> **"稳定化法"**：通过`@Stable`/`@Immutable`标记、使用`derivedStateOf`派生状态、`remember`缓存计算、`key`控制重组粒度，确保Compose只重组真正变化的部分。核心是"让Compose知道什么变了，什么没变"。

### 案例 11：Bitmap 内存优化 — 采样加载与复用

**问题背景**：加载大图导致OOM，Bitmap占用内存过高。

**优化方案**：

```kotlin
object BitmapOptimizer {
    fun decodeSampledBitmap(
        resolver: ContentResolver, uri: Uri, reqWidth: Int, reqHeight: Int
    ): Bitmap? {
        return resolver.openFileDescriptor(uri, "r")?.use { fd ->
            val options = BitmapFactory.Options().apply {
                inJustDecodeBounds = true
                BitmapFactory.decodeFileDescriptor(fd.fileDescriptor, null, this)
                inSampleSize = calculateInSampleSize(this, reqWidth, reqHeight)
                inJustDecodeBounds = false
                inPreferredConfig = Bitmap.Config.RGB_565
                inMutable = true
            }
            BitmapFactory.decodeFileDescriptor(fd.fileDescriptor, null, options)
        }
    }

    private fun calculateInSampleSize(options: BitmapFactory.Options, reqW: Int, reqH: Int): Int {
        val (h, w) = options.run { outHeight to outWidth }
        var size = 1
        while (h / size >= reqH * 2 || w / size >= reqW * 2) size *= 2
        return size
    }

    val bitmapCache = object : LruCache<String, Bitmap>(
        (Runtime.getRuntime().maxMemory() / 8).toInt()
    ) {
        override fun sizeOf(key: String, value: Bitmap) = value.byteCount / 1024
        override fun entryRemoved(evicted: Boolean, key: String, old: Bitmap, new: Bitmap?) {
            if (evicted && !old.isRecycled) old.recycle()
        }
    }
}
```

**优化效果**：内存占用减少60%，OOM崩溃率降低90%。

**方法论归纳**：
> **"按需加载法"**：按View尺寸采样加载、使用低色深格式（RGB_565）、复用Bitmap内存、LruCache缓存管理。核心是"只加载用户能看到的大小，用完后立即释放"。

### 案例 13：大对象内存管理 — Native 内存与 Java 堆平衡

**问题背景**：图片处理导致Java堆内存不足，频繁GC。

**优化方案**：

```kotlin
class ImageProcessor {
    fun processLargeImage(input: Bitmap): Bitmap {
        val rs = RenderScript.create(context)
        val inputAlloc = Allocation.createFromBitmap(rs, input)
        val outputAlloc = Allocation.createTyped(rs, inputAlloc.type)
        val script = ScriptIntrinsicBlur.create(rs, Element.U8_4(rs))
        script.setRadius(25f)
        script.setInput(inputAlloc)
        script.forEach(outputAlloc)
        val output = Bitmap.createBitmap(input.width, input.height, input.config)
        outputAlloc.copyTo(output)
        rs.destroy()  // 及时释放Native资源
        return output
    }

    private val byteArrayPool = object : Pool<ByteArray>(10) {
        override fun createNew() = ByteArray(1024 * 1024)
        override fun reset(instance: ByteArray) { instance.fill(0) }
    }
}

abstract class Pool<T>(private val maxSize: Int) {
    private val pool = ArrayDeque<T>(maxSize)
    fun acquire(): T = pool.removeFirstOrNull() ?: createNew()
    fun release(instance: T) {
        if (pool.size < maxSize) { reset(instance); pool.addLast(instance) }
    }
    abstract fun createNew(): T
    abstract fun reset(instance: T)
}
```

**优化效果**：Java堆内存占用减少50%，GC频率降低70%。

**方法论归纳**：
> **"分层存储法"**：大对象优先使用Native内存（RenderScript/OpenGL）或对象池，避免Java堆压力；文件处理采用流式分块。核心是"Java堆是稀缺资源，能不用就不用"。

```kotlin
            TRIM_MEMORY_MODERATE, TRIM_MEMORY_COMPLETE -> {

        dataCache.keys.filter { !it.startsWith(currentPage) }
            .forEach { dataCache.remove(it) }
    }
}
```

### 案例 20：WakeLock 优化 — 避免不当持有

**问题背景**：下载任务持有WakeLock不释放，导致屏幕关闭后CPU持续运行。

```kotlin
class SafeDownloadManager(private val context: Context) {
    private val powerManager = context.getSystemService(Context.POWER_SERVICE) as PowerManager
    private var wakeLock: PowerManager.WakeLock? = null

    fun startDownload(url: String) {
        wakeLock = powerManager.newWakeLock(PowerManager.PARTIAL_WAKE_LOCK, "MyApp:DownloadWakeLock")
            .apply { acquire(10 * 60 * 1000L) }  // 最多10分钟
        downloadScope.launch {
            try { performDownload(url) } finally { releaseWakeLock() }
        }
    }

    class DownloadWorker(context: Context, params: WorkerParameters) : Worker(context, params) {
        override fun doWork(): Result {
            return performDownload()  // WorkManager自动管理WakeLock
        }
    }

    fun startWifiDownload(url: String) {
        val wifiManager = context.applicationContext.getSystemService(Context.WIFI_SERVICE) as WifiManager
        val wifiLock = wifiManager.createWifiLock(WifiManager.WIFI_MODE_FULL_HIGH_PERF, "DownloadWifiLock")
        wifiLock.acquire()
        try { performDownload(url) } finally { if (wifiLock.isHeld) wifiLock.release() }
    }

    private fun releaseWakeLock() {
        wakeLock?.let { if (it.isHeld) it.release(); wakeLock = null }
    }
}
```

### 案例 29：弱网环境优化 — 超时重试与降级策略

**问题背景**：弱网环境下请求频繁失败，用户体验差。

**优化方案**：

```kotlin
class WeakNetworkOptimizer {
    private val client = OkHttpClient.Builder()
        .connectTimeout(getDynamicTimeout(), TimeUnit.SECONDS)
        .readTimeout(getDynamicTimeout() * 2, TimeUnit.SECONDS)
        .build()

    fun getNetworkQuality(): NetworkQuality {
        val startTime = System.currentTimeMillis()
        return try {
            val response = client.newCall(Request.Builder()
                .url("https://www.google.com/generate_204").build()).execute()
            val rtt = System.currentTimeMillis() - startTime
            when {
                rtt < 100 -> NetworkQuality.EXCELLENT
                rtt < 300 -> NetworkQuality.GOOD
                rtt < 1000 -> NetworkQuality.POOR
                else -> NetworkQuality.BAD
            }
        } catch (e: Exception) { NetworkQuality.BAD }
    }

    suspend fun <T> requestWithRetry(apiCall: suspend () -> T, maxRetries: Int = 3): Result<T> {
        var lastException: Exception? = null
        repeat(maxRetries) { attempt ->
            try {
                if (attempt > 0) delay((1000 * 2.0.pow(attempt - 1)).toLong())
                return Result.success(apiCall())
            } catch (e: Exception) { lastException = e }
        }
        return Result.failure(lastException ?: Exception("Unknown error"))
    }

    class DegradedApiService(private val normalApi: ApiService, private val cache: LocalCache) : ApiService {
        override suspend fun getHomeData(): HomeData {
            return try { normalApi.getHomeData().also { cache.saveHomeData(it) } }
            catch (e: Exception) { cache.getHomeData() ?: throw e }
        }
        override suspend fun getHighResImage(): ByteArray {
            return try { normalApi.getHighResImage() } catch (e: Exception) { normalApi.getLowResImage() }
        }
    }
}
```

**优化效果**：弱网环境下请求成功率从40%提升至85%。

**方法论归纳**：
> **"自适应降级法"**：根据网络质量动态调整超时、指数退避重试、失败时降级到缓存/低质量资源。核心是"弱网不是不服务，而是提供降级的服务"。

### 案例 30：图片加载优化 — 三级缓存与智能裁剪

**问题背景**：图片加载慢、内存占用高、流量消耗大。

**优化方案**：

```kotlin
class ImageOptimizationConfig {
    fun createOptimizedLoader(context: Context): ImageLoader {
        return ImageLoader.Builder(context)
            .memoryCache {
                MemoryCache.Builder(context)
                    .maxSizePercent(0.25)
                    .build()
            }
            .diskCache {
                DiskCache.Builder()
                    .directory(context.cacheDir.resolve("image_cache"))
                    .maxSizeBytes(512L * 1024 * 1024)
                    .build()
            }
            .okHttpClient {
                OkHttpClient.Builder()
                    .cache(Cache(context.cacheDir.resolve("http_cache"), 100L * 1024 * 1024))
                    .build()
            }
            .crossfade(true)
            .placeholder(R.drawable.placeholder)
            .error(R.drawable.error)
            .build()
    }

    fun loadImage(imageView: ImageView, url: String) {
        imageView.load(url) {
            size(ViewSizeResolver(imageView))
            transformations(CircleCropTransformation())
            if (networkQuality == NetworkQuality.POOR) quality(80)
        }
    }

    fun loadProgressive(imageView: ImageView, url: String) {
        imageView.load(url) {
            placeholderMemoryCacheKey(url)
            crossfade(300)
            thumbnail { data("$url?w=100") }
        }
    }
}
```

**优化效果**：图片加载速度提升50%，内存占用减少60%，流量消耗减少40%。

**方法论归纳**：
> **"三级缓存法"**：内存缓存（LruCache）→ 磁盘缓存（DiskCache）→ 网络缓存（HTTP Cache），每层都有淘汰策略；加载时按View尺寸裁剪、按网络质量调整质量、优先使用WebP。核心是"能不下载就不下载，能下载小的就不下载大的"。

## 八、性能优化的"黄金法则"

### 法则一：测量先行

> **"没有测量就没有优化"**
> 
> 使用Profiler、Systrace、Macrobenchmark等工具定位真正瓶颈，避免盲目优化。

### 法则二：二八定律

> **"80%的性能问题来自20%的代码"**
> 
> 聚焦高频路径（启动、首页、核心流程），不要试图优化所有代码。

### 法则三：渐进优化

> **"先正确，再优化；先局部，再全局"**
> 
> 不要一次性重写整个架构，逐步替换、逐步验证。

### 法则四：权衡取舍

> **"性能、功能、复杂度，三者不可兼得"**
> 
> 根据产品阶段和用户价值做出合理取舍，不要为1%的性能提升增加50%的复杂度。

### 法则五：持续监控

> **"优化是过程，不是结果"**
> 
> 建立性能基线，每次发版对比，防止性能退化。
