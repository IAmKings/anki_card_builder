## 一、架构模式基础 (1-10)

### 1. MVC、MVP、MVVM、MVI 四种架构模式的核心区别是什么?

| 特性 | MVC | MVP | MVVM | MVI |
|------|-----|-----|------|-----|
| **数据流向** | 双向 | 双向 | 双向绑定 | 单向数据流 |
| **View与Model关系** | View可直接访问Model | 完全隔离 | 通过ViewModel间接 | 完全隔离 |
| **通信方式** | 直接引用/通知 | 接口回调 | 数据绑定/观察者 | Intent驱动 |
| **可测试性** | 差 | 好 | 很好 | 极好 |
| **代码量** | 少 | 中 | 中 | 多 |
| **适用场景** | 小型项目 | 中型项目 | 大型项目 | 复杂交互/Compose |

**核心演进逻辑**：从MVC到MVI，本质是**关注点分离**的不断深化，Controller/Presenter/ViewModel逐渐瘦身View，MVI进一步通过不可变状态解决了MVVM中状态分散的问题。

### 2. 为什么Android中MVC容易导致Activity臃肿?

Android的Activity天然承担了**View和Controller双重角色**：

1. **View职责**：`setContentView()`、`findViewById()`、UI更新
2. **Controller职责**：生命周期管理、事件处理、业务逻辑、数据获取

```kotlin
// 典型MVC反模式
class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        // View职责
        val textView = findViewById<TextView>(R.id.text)
        val button = findViewById<Button>(R.id.button)

        // Controller + Model职责混杂
        button.setOnClickListener {
            lifecycleScope.launch {
                val response = RetrofitClient.api.fetchData()  // 网络请求
                val processed = processData(response)           // 业务逻辑
                textView.text = processed                       // UI更新
            }
        }
    }
}
```

**后果**：Activity代码数千行，难以测试、难以维护。

### 3. MVP如何解决MVC的问题?它的缺点是什么?

**解决方案**：
- 引入**Presenter**作为View和Model的中介
- View和Model完全隔离，通过接口通信
- Activity只负责UI展示和事件转发

```kotlin
// Contract定义接口
interface LoginContract {
    interface View {
        fun showLoading()
        fun hideLoading()
        fun showError(message: String)
        fun navigateToHome()
    }

    interface Presenter {
        fun login(username: String, password: String)
        fun onDestroy()
    }
}

// Presenter实现
class LoginPresenter(private val view: LoginContract.View) : LoginContract.Presenter {
    override fun login(username: String, password: String) {
        view.showLoading()
        // 调用Model层...
    }
}
```

**缺点**：
1. **接口爆炸**：View和Presenter交互复杂时接口膨胀
2. **Presenter臃肿**：业务逻辑集中导致Presenter过重
3. **内存泄漏风险**：Presenter持有View引用，需手动管理生命周期
4. **双向依赖**：View变化需同步修改Presenter

### 4. MVVM中ViewModel如何做到配置变更后数据不丢失?

通过`ViewModelStore`和`ViewModelProvider`机制：

```kotlin
// 配置变更时（如旋转屏幕）
// Activity销毁 → onRetainNonConfigurationInstance()保留ViewModelStore
// Activity重建 → getLastNonConfigurationInstance()恢复ViewModelStore

class MyViewModel : ViewModel() {
    private val _data = MutableLiveData<String>()
    val data: LiveData<String> = _data

    init {
        // 只在首次创建时执行，配置变更不重新初始化
        loadData()
    }
}
```

**原理**：
- `ViewModel`实例存储在`ViewModelStore`中
- `ViewModelStore`通过`NonConfigurationInstances`在配置变更时保留
- 新Activity通过`ViewModelProvider`获取已存在的实例

### 5. MVI的"单向数据流"具体是如何工作的?

```
User Action → Intent → ViewModel → Reducer → New State → View Recomposition
```

```kotlin
// 1. 定义不可变状态
data class UiState(
    val isLoading: Boolean = false,
    val users: List<User> = emptyList(),
    val error: String? = null
)

// 2. 定义Intent（用户意图）
sealed class UiIntent {
    object LoadUsers : UiIntent()
    data class SearchUser(val query: String) : UiIntent()
    data class ClickUser(val userId: String) : UiIntent()
}

// 3. ViewModel处理
class UserViewModel : ViewModel() {
    private val _state = MutableStateFlow(UiState())
    val state: StateFlow<UiState> = _state.asStateFlow()

    fun processIntent(intent: UiIntent) {
        when (intent) {
            is UiIntent.LoadUsers -> loadUsers()
            is UiIntent.SearchUser -> searchUsers(intent.query)
            is UiIntent.ClickUser -> navigateToDetail(intent.userId)
        }
    }

    private fun loadUsers() {
        _state.update { it.copy(isLoading = true) }
        viewModelScope.launch {
            try {
                val users = repository.getUsers()
                _state.update { it.copy(isLoading = false, users = users) }
            } catch (e: Exception) {
                _state.update { it.copy(isLoading = false, error = e.message) }
            }
        }
    }
}
```

**优势**：状态可预测、可回溯、时间旅行调试。

### 6. VIPER架构相比MVVM多了哪些层次?适用什么场景?

VIPER将职责进一步细分：

| 层次 | 职责 | 对应MVVM |
|------|------|----------|
| **View** | UI展示和用户事件转发 | View |
| **Presenter** | 视图逻辑、格式化数据 | ViewModel（部分） |
| **Interactor** | 业务逻辑、用例执行 | ViewModel（部分） |
| **Entity** | 数据模型 | Model |
| **Router** | 页面导航和模块跳转 | 通常缺失 |

```swift
// iOS示例
class UserRouter: UserRouterProtocol {
    static func createModule() -> UIViewController {
        let view = UserViewController()
        let presenter = UserPresenter()
        let interactor = UserInteractor()
        let router = UserRouter()

        view.presenter = presenter
        presenter.view = view
        presenter.interactor = interactor
        presenter.router = router
        interactor.presenter = presenter
        router.viewController = view

        return view
    }

    func navigateToDetail(userId: String) {
        let detailVC = DetailRouter.createModule(userId: userId)
        viewController?.navigationController?.pushViewController(detailVC, animated: true)
    }
}
```

**适用场景**：超大型项目、多团队协作、iOS原生开发（Android较少使用）。

### 7. 在Compose时代，为什么MVI比MVVM更合适?

1. **声明式UI天然适配单向数据流**：Compose的`@Composable`函数是"State → UI"的纯函数映射
2. **重组机制**：Compose的重组基于状态变化，MVI的单一State对象正好匹配
3. **状态一致性**：避免MVVM中多个LiveData/StateFlow导致的状态不同步
4. **副作用管理**：Compose的`LaunchedEffect`/`SideEffect`与MVI的Intent处理完美配合

```kotlin
@Composable
fun UserScreen(viewModel: UserViewModel = hiltViewModel()) {
    val state by viewModel.state.collectAsStateWithLifecycle()

    when {
        state.isLoading -> LoadingIndicator()
        state.error != null -> ErrorMessage(state.error)
        else -> UserList(
            users = state.users,
            onUserClick = { viewModel.processIntent(UiIntent.ClickUser(it)) }
        )
    }
}
```

### 8. 如何评价一个架构设计的好坏?

**核心标准**：

1. **关注点分离（Separation of Concerns）**：各层职责清晰，不越界
2. **可测试性（Testability）**：业务逻辑可脱离Android框架单元测试
3. **可维护性（Maintainability）**：修改一处不影响全局
4. **可扩展性（Scalability）**：新增功能不破坏现有结构
5. **可复用性（Reusability）**：组件可在不同项目复用
6. **团队适应性**：符合团队技术水平和协作方式

**反模式警示**：
- 过度设计（简单项目用VIPER）
- 教条主义（为了模式而模式）
- 忽视性能（过度抽象导致性能损耗）

### 9. 在实际项目中，如何渐进式地从MVP迁移到MVVM?

**迁移策略**：

1. **阶段一：共存期**
   - 新模块使用MVVM，旧模块保持MVP
   - 提取公共Repository层

2. **阶段二：中间层**
   - 将MVP的Presenter改造为ViewModel
   - 接口回调改为LiveData/StateFlow

3. **阶段三：完全迁移**
   - 移除所有Presenter和Contract接口
   - 统一使用DataBinding/ViewBinding

```kotlin
// 兼容层：Presenter风格的ViewModel
class HybridViewModel(
    private val repository: UserRepository
) : ViewModel() {
    // 兼容旧代码的回调方式
    private val _callback = MutableLiveData<PresenterCallback>()

    // 新代码使用的StateFlow
    private val _state = MutableStateFlow(UiState())
    val state: StateFlow<UiState> = _state.asStateFlow()

    fun loadData() {
        viewModelScope.launch {
            _callback.value = PresenterCallback.ShowLoading
            _state.update { it.copy(isLoading = true) }

            val result = repository.fetch()

            _callback.value = PresenterCallback.HideLoading
            _state.update { it.copy(isLoading = false, data = result) }
        }
    }
}
```

### 10. Coordinator模式在移动端架构中的作用是什么?

**解决导航逻辑分散问题**：将页面跳转从ViewController/Activity中抽离。

```swift
// iOS Coordinator
protocol Coordinator {
    var childCoordinators: [Coordinator] { get set }
    var navigationController: UINavigationController { get set }
    func start()
}

class AppCoordinator: Coordinator {
    var childCoordinators = [Coordinator]()
    var navigationController: UINavigationController

    func start() {
        let loginCoordinator = LoginCoordinator(navigationController: navigationController)
        childCoordinators.append(loginCoordinator)
        loginCoordinator.start()
    }

    func didLogin() {
        childCoordinators.removeAll()
        let mainCoordinator = MainCoordinator(navigationController: navigationController)
        childCoordinators.append(mainCoordinator)
        mainCoordinator.start()
    }
}
```

**Android替代方案**：Navigation Component + Deep Link、ARouter、自研路由框架。

## 二、Clean Architecture与分层设计 (11-20)

### 11. Clean Architecture的核心思想是什么?

由Robert C. Martin提出，核心思想是**依赖规则**：

```
         ┌─────────────┐
         │   Entities  │  ← 最内层，业务核心
         │  (Domain)   │
         ├─────────────┤
         │  Use Cases  │  ← 业务逻辑
         ├─────────────┤
         │ Interface   │  ← 接口适配器（Presenter/Repository）
         │  Adapters   │
         ├─────────────┤
         │ Frameworks  │  ← 最外层，UI/数据库/网络
         │    & Drivers│
         └─────────────┘
```

**依赖规则**：内层不依赖外层，外层依赖内层。所有依赖指向中心。

### 12. 在Android中如何实践Clean Architecture的三层结构?

```
app/
├── presentation/          # UI层（ViewModel/Activity/Fragment/Compose）
│   ├── viewmodel/
│   ├── ui/
│   └── mapper/          # DTO → UI Model转换
├── domain/              # 领域层（核心业务逻辑）
│   ├── model/           # 实体类（User、Order）
│   ├── repository/      # Repository接口
│   └── usecase/         # 用例（GetUserUseCase、LoginUseCase）
└── data/                # 数据层（Repository实现）
    ├── local/           # Room/SharedPreferences
    ├── remote/          # Retrofit/OkHttp
    └── repository/        # RepositoryImpl
```

```kotlin
// Domain层（不依赖Android框架）
interface UserRepository {
    suspend fun getUser(id: String): User
}

class GetUserUseCase(private val repository: UserRepository) {
    suspend operator fun invoke(id: String): Result<User> = runCatching {
        repository.getUser(id)
    }
}

// Data层（依赖Domain层）
class UserRepositoryImpl(
    private val remote: UserRemoteDataSource,
    private val local: UserLocalDataSource
) : UserRepository {
    override suspend fun getUser(id: String): User {
        return local.getUser(id) ?: remote.fetchUser(id).also { local.saveUser(it) }
    }
}
```

### 13. UseCase层存在的意义是什么?什么时候可以省略?

**存在的意义**：
1. **封装业务逻辑**：一个用例一个类，职责单一
2. **跨ViewModel复用**：多个ViewModel可共享UseCase
3. **可测试**：纯Kotlin代码，无Android依赖
4. **表达意图**：类名即业务操作

```kotlin
class ValidateEmailUseCase {
    operator fun invoke(email: String): ValidationResult {
        return when {
            email.isBlank() -> ValidationResult.Error.Empty
            !email.contains("@") -> ValidationResult.Error.Invalid
            else -> ValidationResult.Success
        }
    }
}
```

**可省略的场景**：
- 简单CRUD操作（Repository直接返回数据）
- 小型项目（过度分层增加复杂度）

### 14. Repository模式如何统一多数据源（网络+本地+缓存）?

```kotlin
class UserRepositoryImpl(
    private val remote: UserRemoteDataSource,
    private val local: UserLocalDataSource,
    private val memoryCache: MemoryCache
) : UserRepository {

    override fun getUsers(): Flow<List<User>> = flow {
        // 1. 先发射本地缓存
        emit(memoryCache.get() ?: local.getUsers())

        // 2. 网络请求
        try {
            val remoteUsers = remote.fetchUsers()
            local.saveUsers(remoteUsers)
            memoryCache.put(remoteUsers)
            emit(remoteUsers)
        } catch (e: Exception) {
            // 网络失败，本地数据已发射，静默处理
        }
    }.flowOn(Dispatchers.IO)

    override suspend fun refreshUsers(): List<User> {
        return remote.fetchUsers().also {
            local.saveUsers(it)
            memoryCache.put(it)
        }
    }
}
```

**策略模式**：
- **Cache-First**：优先本地，后台刷新（适合列表）
- **Network-First**：优先网络，失败降级本地（适合支付）
- **Cache-Only**：纯离线（适合已下载内容）

### 15. 什么是SOLID原则?在客户端架构中如何应用?

| 原则 | 含义 | 客户端应用 |
|------|------|----------|
| **S**ingle Responsibility | 单一职责 | Activity只做UI，ViewModel只做业务逻辑 |
| **O**pen/Close | 开闭原则 | 新增功能通过扩展而非修改现有代码 |
| **L**iskov Substitution | 里氏替换 | 子类ViewModel可替换父类不影响功能 |
| **I**nterface Segregation | 接口隔离 | Repository接口只暴露必要方法 |
| **D**ependency Inversion | 依赖倒置 | 依赖抽象（接口）而非具体实现 |

// Data层（可替换为Glide/Picasso/Coil）

### 16. 如何设计一个可测试的客户端架构?

**核心策略**：

1. **依赖注入**：通过构造函数注入依赖，便于Mock
2. **接口抽象**：Repository、DataSource都定义接口
3. **避免Android框架依赖**：业务逻辑不依赖Context/Activity
4. **测试替身**：使用Fake/Mock/Stub替代真实依赖

```kotlin
// 可测试的ViewModel
class LoginViewModel(
    private val loginUseCase: LoginUseCase,      // 接口注入
    private val validator: InputValidator,        // 接口注入
    private val dispatcher: CoroutineDispatcher = Dispatchers.Main  // 可替换调度器
) : ViewModel() {

    fun login(email: String, password: String) {
        viewModelScope.launch(dispatcher) {
            when (val result = loginUseCase(email, password)) {
                is Result.Success -> _navigation.value = Route.Home
                is Result.Error -> _error.value = result.message
            }
        }
    }
}

// 测试
@Test
fun `login with valid credentials navigates to home`() = runTest {
    val loginUseCase = mockk<LoginUseCase>()
    coEvery { loginUseCase(any(), any()) } returns Result.Success

    val viewModel = LoginViewModel(loginUseCase, FakeValidator(), StandardTestDispatcher())
    viewModel.login("test@test.com", "password")

    assertEquals(Route.Home, viewModel.navigation.value)
}
```

### 19. 如何处理跨层异常传递?

**策略**：每层转换异常为适合该层的错误类型。

```kotlin
// Data层异常 → Domain层Result
sealed class NetworkError {
    object NoConnection : NetworkError()
    object Unauthorized : NetworkError()
    data class ServerError(val code: Int) : NetworkError()
}

class UserRepositoryImpl(private val api: UserApi) : UserRepository {
    override suspend fun getUser(id: String): Result<User> = try {
        Result.Success(api.fetchUser(id).toDomain())
    } catch (e: IOException) {
        Result.Error(NetworkError.NoConnection)
    } catch (e: HttpException) {
        when (e.code()) {
            401 -> Result.Error(NetworkError.Unauthorized)
            else -> Result.Error(NetworkError.ServerError(e.code()))
        }
    }
}

// Domain层 → UI层
class GetUserUseCase(private val repository: UserRepository) {
    suspend operator fun invoke(id: String): UiResult<User> {
        return when (val result = repository.getUser(id)) {
            is Result.Success -> UiResult.Success(result.data)
            is Result.Error -> when (result.error) {
                NetworkError.NoConnection -> UiResult.Error("请检查网络连接")
                NetworkError.Unauthorized -> UiResult.Error("登录已过期，请重新登录")
                is NetworkError.ServerError -> UiResult.Error("服务器繁忙，请稍后再试")
            }
        }
    }
}
```

### 20. 模块化、组件化、插件化的区别是什么?

| **拆分粒度** | 按功能/层拆分模块 | 按业务拆分独立组件 | 动态下发功能模块 |

## 三、状态管理与响应式编程 (21-30)

### 21. 为什么移动端需要专门的状态管理方案?

**问题背景**：
- Android配置变更（旋转屏幕）导致Activity重建
- 多Fragment共享状态困难
- 异步回调导致状态不一致

**状态管理核心需求**：
1. **生命周期感知**：自动处理配置变更
2. **作用域控制**：页面级/全局级/用户级状态
3. **线程安全**：主线程更新UI
4. **可观察性**：状态变化自动通知UI

### 22. LiveData、StateFlow、BehaviorSubject的区别是什么?

| 特性 | LiveData | StateFlow | BehaviorSubject |
|------|----------|-----------|-----------------|
| **平台** | Android专用 | Kotlin Multiplatform | RxJava |
| **生命周期感知** | ✓ 自动 | ✗ 需配合`repeatOnLifecycle` | ✗ 需手动管理 |
| **初始值** | 可无 | 必须有 | 必须有 |

### 30. 什么是时间旅行调试（Time Travel Debugging）?MVI如何实现?

**原理**：记录所有状态变更历史，可回放、暂停、跳转。

```kotlin
class TimeTravelStore<State, Action>(
    initialState: State,
    private val reducer: (State, Action) -> State
) {
    private val _history = mutableListOf(initialState)
    private var _currentIndex = 0

    val currentState: State get() = _history[_currentIndex]

    fun dispatch(action: Action) {
        val newState = reducer(currentState, action)
        // 删除当前位置之后的历史（分支处理）
        if (_currentIndex < _history.lastIndex) {
            _history.subList(_currentIndex + 1, _history.size).clear()
        }
        _history.add(newState)
        _currentIndex++
    }

    fun travelTo(index: Int) {
        _currentIndex = index.coerceIn(0, _history.lastIndex)
    }

    fun undo() = travelTo(_currentIndex - 1)
    fun redo() = travelTo(_currentIndex + 1)
}
```

            navController.popBackStack()
        }
    }

### 48. 什么是响应式架构中的"命令式外壳，声明式核心"?

**概念**：底层使用命令式代码处理副作用，上层暴露声明式API。

```kotlin
// 声明式核心：纯函数，无副作用
class CartCalculator {
    fun calculateTotal(items: List<CartItem>): BigDecimal {
        return items.sumOf { it.price * it.quantity.toBigDecimal() }
    }

    fun applyDiscount(total: BigDecimal, discount: Discount): BigDecimal {
        return when (discount) {
            is Discount.Percentage -> total * (1 - discount.rate)
            is Discount.FixedAmount -> (total - discount.amount).coerceAtLeast(BigDecimal.ZERO)
        }
    }
}

// 命令式外壳：处理副作用（网络、数据库、UI）
class CartViewModel(
    private val calculator: CartCalculator,
    private val repository: CartRepository
) : ViewModel() {
    fun checkout() {
        viewModelScope.launch {
            // 命令式：获取数据
            val items = repository.getCartItems()

            // 声明式：计算逻辑
            val total = calculator.calculateTotal(items)
            val finalPrice = calculator.applyDiscount(total, currentDiscount)

            // 命令式：提交订单
            repository.placeOrder(finalPrice)
        }
    }
}
```

### 49. 如何设计一个可扩展的错误处理架构?

```kotlin
// 统一错误类型
sealed class AppError {
    abstract val message: String

    data class Network(override val message: String, val code: Int? = null) : AppError()
    data class Validation(override val message: String, val field: String? = null) : AppError()
    data class Business(override val message: String, val errorCode: String) : AppError()
    data class Unknown(override val message: String, val throwable: Throwable? = null) : AppError()
}

// 错误处理器链
interface ErrorHandler {
    suspend fun handle(error: AppError): Boolean  // 返回true表示已处理
}

class NetworkErrorHandler(private val context: Context) : ErrorHandler {
    override suspend fun handle(error: AppError): Boolean {
        if (error !is AppError.Network) return false
        if (error.code == 401) {
            // 跳转登录页
            context.startActivity(Intent(context, LoginActivity::class.java))
            return true
        }
        return false
    }
}

class ToastErrorHandler(private val context: Context) : ErrorHandler {
    override suspend fun handle(error: AppError): Boolean {
        Toast.makeText(context, error.message, Toast.LENGTH_SHORT).show()
        return true
    }
}

// 使用
class ErrorHandlerChain(private val handlers: List<ErrorHandler>) {
    suspend fun handle(error: AppError) {
        for (handler in handlers) {
            if (handler.handle(error)) break
        }
    }
}
```

### 50. 未来客户端架构的发展趋势是什么?

1. **跨平台统一**：Compose Multiplatform、KMP实现Android/iOS/Web共享业务逻辑
2. **声明式UI成为主流**：Compose、SwiftUI取代命令式UI
3. **Server-Driven UI**：服务端下发UI配置，客户端动态渲染
4. **AI辅助架构**：AI生成样板代码、自动重构、智能代码审查
5. **边缘计算**：部分业务逻辑下沉到端侧，减少服务端依赖
6. **微前端化**：大型App拆分为独立子应用，通过容器动态加载
7. **响应式增强**：StateFlow + Compose成为标准状态管理方案
8. **性能优先架构**：Baseline Profiles、预编译、懒加载成为标配

## 总结

客户端架构设计的核心在于**平衡**：
- 简单与复杂
- 性能与可维护性
- 规范与灵活
- 短期交付与长期演进

没有最好的架构，只有**最适合当前团队、当前项目、当前阶段**的架构。建议从实际需求出发，渐进式演进，避免过度设计。
